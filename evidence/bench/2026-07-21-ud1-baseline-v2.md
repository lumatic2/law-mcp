- 일자: 2026-07-21 · 세트 `bench/golden-v2.json` dev 25건 · 파이프라인 = LB5 완료 상태 + UD0 핫픽스
- 원자료: `2026-07-21-ud1-baseline-v2.json` · `-repeat.json` · `-batch2.json` · `2026-07-21-probe-fresh.json`

## 기준선

| 지표 | 신 세트(golden-v2) dev | 구 세트(golden) dev |
|---|---|---|
| **blind recall@3** | **48.0%** | 76.0% |
| blind recall@1 | 32.0% | 64.0% |
| 도메인별 recall@3 | 민사 100% · 노동 40% · 형사 40% · 세무 40% · 행정 20% | — |

**28%p 차이는 도구가 나빠진 게 아니라 구 세트가 소진됐다는 증거다.** 구 dev 25건은 LB2·LB5 가
그 25건을 보고 파라미터를 골랐다(F5). 76% 는 그 셋에 맞춘 값이고, 48% 가 **처음 보는 질의에서의
실제 실력**이다. 이 격차 자체가 새 세트를 만든 이유를 사후 정당화한다.

행정 20% 는 신규 도메인이라 예상 범위다 — 이 레포는 행정법 질의를 한 번도 측정한 적이 없다.

## ★ 이 step 의 핵심 발견 — 노이즈가 아니라 **드리프트**다

계획은 "반복 측정으로 노이즈를 정량화한다"였다. 실제로 잰 결과 **문제는 노이즈가 아니었다.**

측정 11회를 시간순으로:

```
배치1 (n=5)  60.0  60.0  60.0  60.0  60.0
배치2 (n=5)  60.0  48.0  48.0  48.0  48.0
단발  (n=1)  48.0
배치3 (n=5)  48.0  48.0  48.0  48.0  48.0
```

**60% 가 6회 연속 → 48% 가 10회 연속. 한 번 내려간 뒤 돌아오지 않았다.** 무작위 흔들림이라면
섞여 나와야 한다. 이건 상류 상태가 바뀐 것이다.

### 무엇이 바뀌었나 — 뒤집힌 쿼리 5건 대조

| 쿼리 | 60% 시점 | 48% 시점 |
|---|---|---|
| 행정심판 청구 기간 | **행정심판법** / 행정심판법 시행령 / 검찰청법 | 검찰청법 / 가사소송법 / 고용보험법 |
| 취소소송 제소기간 | **행정소송법** / 관세법 / 상법 | 상법 / 건축법 / 중재법 |
| 경영상 이유에 의한 해고 요건 | 상법 / **근로기준법** / 소득세법 시행령 | 전략적투자 특별법 / 방위사업법 / 상법 |
| 매입세액 불공제 대상 | 부가가치세법 시행령 / **부가가치세법** / 시행규칙 | 부가가치세법 시행규칙 (**1건만**) |
| 지상권 존속기간 | 국회법 / 도로법 / 도시개발법 | 특허법 / **민법** / 국회법 |

공통점이 뚜렷하다: **법령명 검색이 0건이 되어 본문검색으로 떨어졌다.** 60% 시점에는 완화 단계가
`행정심판 청구 기간` → 행정심판법을 찾아냈는데, 48% 시점에는 그 단계가 0건을 반환하고 사다리가
본문검색까지 내려간다. 본문검색은 가나다순이라(ib3 확인) 결과가 노이즈가 된다.
반환 건수가 3건 → 1건으로 줄어든 사례도 같은 방향을 가리킨다.

### 유력한 원인과, 그 때문에 이 수치를 어떻게 다뤄야 하나

가장 그럴듯한 설명은 **상류 쓰로틀링**이다. 오늘 이 세션에서 법제처 API 를 수천 회 호출했고,
그 뒤로 검색 채널이 빈약한 응답을 주기 시작했다. 그렇다면:

- **48.0% 는 "쓰로틀 걸린 상태의 실력"일 수 있고, 평상시 사용자는 60% 를 볼 수 있다.**
- 어느 쪽이 참인지는 **호출을 쉬고 다시 재야** 안다. 이 문서는 두 수치를 모두 남긴다.

## 채택 문턱 — 계획이 세운 2σ 는 쓸 수 없다

계획은 "A/B 차이가 2σ 를 넘으면 채택"으로 정했다. 실측이 그 규칙을 무효화한다:

- 배치3 의 σ 는 **0.0%p** 다. 문턱이 0 이면 아무 차이나 이득으로 인정된다 — 무의미하다.
- 배치2 의 σ 는 5.4%p, 문턱 10.7%p. 같은 코드인데 배치마다 문턱이 0 과 10.7 사이에서 널뛴다.

**연속 실행의 표준편차는 드리프트를 못 잡는다.** 드리프트는 회차 사이가 아니라 배치 사이에서
일어나기 때문이다. 그리고 A/B 를 순차로 재면(A 5회 → B 5회) 드리프트가 통째로 **뒤에 잰 쪽의
불이익**으로 들어간다 — 정확히 우리가 UD2 에서 하려던 일이다.

### 그래서 UD2 의 A/B 는 이 규약을 따른다 (이 step 이 확정하는 것)

1. **교차 측정 필수.** A 배치 → B 배치가 아니라 **쿼리마다 A·B 를 연달아** 잰다. 드리프트가
   양쪽에 똑같이 걸리게 만든다.
2. **판정은 집계가 아니라 쿼리 단위 승패로 한다.** 같은 시점·같은 쿼리에서 A 가 맞고 B 가 틀렸는지를
   센다. 이건 드리프트에 영향받지 않는다.
3. **채택 문턱 = 새로 깨지는 쿼리 0 + 순 이득 ≥3건**(25건 중). 2σ 문턱은 폐기한다 — 위 이유로
   신뢰할 수 없다.
4. 절대 수치를 인용할 때는 **측정 시각과 그 세션의 누적 호출량**을 함께 적는다.
