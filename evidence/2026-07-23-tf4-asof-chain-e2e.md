# TF4 검색→시점조회 체인 실 MCP 관측

> 2026-07-23 · milestone TF4 step-2 · horizon `trap-free`

## 결함 (수리 전)

`getLawArticle` 이 `this.resolveAsOfVersion(lawId, options.asOf)` 로 **법령ID 를 이름 자리에**
넘겼다(`src/providers/lawgo-provider.ts`). 받는 쪽은 `fetchLawVersions(lawName)` 을 호출하는데,
그 조회는 이름으로 질의하고 `법령명한글` 정확일치로 거른다 — 숫자 ID 를 넣으면 0건이 되어
`LAW_VERSION_NOT_FOUND` 로 끊긴다.

`search_law` 는 `law_id` 를 준다. 그래서 **에이전트가 정상 경로로 쓰면 반드시 끊겼다.**
서버 instructions 는 "세금·연도가 걸린 질문은 `as_of` 를 쓰라"고 지시하고 있었다.

## 수리

`resolveLawName(input, resolvedLawId)` 추가 — 숫자 ID 면 본문 조회(`target=law&ID=`)로
`법령명_한글` 을 받아 온다. **이름을 추측하지 않는다.** 이름을 못 받으면 시점 경로로 들어가지
않는다(다시 ID 가 이름 자리에 들어가는 것을 막는다).

## 실 MCP 표면 관측

```
$ npx tsx src/asof-chain-smoke.ts

① search_law('근로소득세액공제')
   → 소득세법 (law_id=001565)
② 받은 law_id 를 그대로 as_of 에 넘긴다
   as_of=2020 → 시행일 20200828 · 본문 627자 · rule="2020년 귀속으로 보고 그 해 12월 31일..."
   as_of=2024 → 시행일 20240701 · 본문 707자 · rule="2024년 귀속으로 보고 그 해 12월 31일..."
③ 시점 없이(현행)
   → 시행일 20260101 · 본문 702자
```

**체인이 이어진다.** `search_law` 가 준 `law_id` 를 사람 손 없이 그대로 넘겨 과거 시점 조문을
받는다. 응답에 `as_of_rule` 이 실린다.

## 실패 검증 — 과거와 현행이 실제로 다른가

시점 해석이 조용히 실패하고 현행으로 대체되면 테스트는 통과하는데 답은 틀린다. 그래서 내용이
실제로 다른지를 본다.

| | 시행일 | 본문 길이 |
|---|---|---|
| `as_of=2020` | 20200828 | 627자 |
| `as_of=2024` | 20240701 | 707자 |
| 현행 | 20260101 | 702자 |

셋 다 다르다 — 시점 해석이 실제로 먹었다. 별도로 소득세법 제59조의4 도 확인했다
(현행 20260101 4022자 vs 2020 시점 20200828 3465자).

해석 불가 시점은 여전히 거절한다:

```
as_of="1800" → 시점 "1800" 을(를) 해석할 수 없다. 연도("2023") 또는 날짜("2023-01-01") 형식으로
               준다. 해석할 수 없는 시점에 현행 법령을 대신 주지 않는다.
```
