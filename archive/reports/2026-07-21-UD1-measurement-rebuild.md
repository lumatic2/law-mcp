# 최종 보고서 — UD1 측정 기반 재건

> 완료: 2026-07-21 · 대상: UD1 (horizon `upstream-delivery`) · 작성: 완료 경계(§B3)

## 1. 문제 정의

이 레포는 **측정 불능 상태**였다. 골든셋 dev 25건은 LB2·LB5 가 그 25건을 보고 파라미터를 골라
소진됐고, holdout 15건은 두 번 열려 죽었다. 게다가 같은 코드로 공식 러너를 두 번 돌리면
72.0%/76.0% 가 나오는데 그 폭이 정량화되지 않아, 4%p 이하 차이는 눈대중으로 처리해 왔다.

사용자가 전수 조사를 근거로 방향을 새로 잡으며 첫 착수 지점을 물었을 때, 선택은 "측정 세트
먼저"였다. 근거는 분명하다 — `aiSearch` 편입처럼 큰 변경일수록, 좋아졌는지 판정할 수단이 먼저
있어야 한다. 없으면 LB1~LB5 의 실수를 반복한다.

## 2. Objective 연결

북극성의 세 축(도달·함정 없음·넓이) 중 어느 것도 이 milestone 이 직접 밀지 않는다. 대신
**그 축들이 움직였는지 말할 수 있는 능력**을 되돌려 놓았다. 앞선 다섯 milestone 에서 가설 6개가
측정으로 죽었고 그게 품질을 지켰다 — 그 판정 능력이 이 레포의 실질적 자산이다.

## 3. 경로

horizon `upstream-delivery` 를 열면서 milestone 3개의 계획을 일괄 작성해 한 번에 승인받았고
(사용자 "ㄱㄱ"), 그 첫 번째가 UD1 이다. 계획한 step 3개(세트 라벨링 → 반복 측정 러너 →
기준선 실측)를 모두 밟았다.

계획과 실제가 갈린 곳이 하나 있다. step-1 에서 라벨을 실 API 로 검증하다 **제품 결함**을 발견해,
계획에 없던 UD0 핫픽스를 앞에 끼워 넣었다(사용자 승인). UD1 은 `src/` 무변경이 원칙이었으므로
별도 changeset·별도 커밋으로 분리했다.

## 4. 구현 결과

세 가지가 생겼다.

**판정에 쓸 수 있는 평가 세트.** 40건(dev 25/holdout 15), 도메인 5종 균등. 구 세트에 없던
**행정·세무**가 들어갔다 — 세무는 이 레포의 최초 소비처인데 정작 골든셋에 한 번도 없었다.
라벨은 추측이 아니라 실 API 응답으로 전건 확인했다.

**흔들림을 말할 수 있는 러너.** `--repeat n` 이 평균·표준편차·범위를 낸다. 1회 측정에서는 편차를
`0` 이 아니라 `n/a` 로 낸다 — 0 은 "흔들리지 않았다"는 거짓 주장이기 때문이다.

**코드가 지키는 홀드아웃 봉인.** 지금까지 규약은 문서에만 있었고, 그래서 세트가 두 번 열려 죽었다.
이제 `--split holdout` 은 거부당하며, 거부 메시지가 사람에게 무엇을 하라고 말한다.

그리고 UD0 로, **`get_law_article` 이 다시 옳은 법의 조문을 반환한다.**

## 5. 이슈와 해결

**이슈 1 — 제품이 조용히 틀린 답을 주고 있었다(near-miss, 최대 발견).**
라벨 검증기가 `근로기준법 제37조` 를 "OK"로 통과시켰는데 돌아온 본문이 보호소년법의 통근취업
조문이었다. `민법` 을 물으면 새마을금고법이, `행정심판법` 을 물으면 광업법이 왔다. 원인은
`resolveLawId` 가 검색 1위를 정확일치 확인 없이 썼고, LB5 부스트가 그 1위를 밀어낸 것이다.
**LB5 의 "새로 깨지는 쿼리 0" 게이트가 이걸 놓친 이유**는 그 측정이 `search_law` 의 *순위*만 보고,
순위가 *다른 도구의 입력*이 된다는 걸 보지 않았기 때문이다. 이름 조회 경로에서 부스트를 끄고
정확일치→prefix 순으로 고르게 고쳤다. 조문 라벨 정확도가 **0/39 → 39/39** 로 바뀌었다.

**이슈 2 — 내가 만든 반복 측정이 거짓 신호를 냈다.**
첫 구현은 provider 를 5회차 내내 재사용해 σ=0.0%p 를 냈다. 캐시가 히트해 결과가 기계적으로
동일해진 것이다. 회차마다 새 인스턴스를 만들도록 고치자 실제 변동이 드러났다.

**이슈 3 — 계획의 채택 문턱이 실측으로 무효화됐다.**
계획은 "차이가 2σ 를 넘으면 채택"이었다. 11회를 재보니 문제는 노이즈가 아니라 **드리프트**였다:
60% 가 6회 연속 → 48% 가 10회 연속, 한 번 내려간 뒤 안 돌아온다. 뒤집힌 쿼리 5건이 전부
법령명 검색 0건 → 본문검색 추락 패턴이었다. 연속 실행의 σ 는 이런 드리프트를 못 잡고(한 배치는
σ=0.0%p → 문턱 0 = 무의미), 더 나쁘게는 A/B 를 순차로 재면 드리프트가 통째로 나중에 잰 쪽의
불이익이 된다. **2σ 문턱을 폐기하고** UD2 규약을 교차 측정 + 쿼리 단위 승패로 바꿨다.

증거: `evidence/bench/2026-07-21-ud1-baseline-v2.md` + 원자료 JSON 4건.

검증: `npm test` **113/113**(94 → 113, 신설 19건). 라벨 실 API 검증
`npm run bench:verify-labels -- --set golden-v2` → 법령명 0 실패 / 조문 0 실패.
기준선 `npm run bench:golden -- --set golden-v2 --split dev --repeat 5` 완주 ×2 배치 →
dev blind recall@3 **48.0%**. `git diff HEAD -- src/` 가 UD1 3 step 에서 빈 것으로 `src/` 무변경 증명.

실패 경로도 실제로 실행했다: 홀드아웃을 플래그 없이 열려다 **거부됨을 관측**했고, 구 세트 쿼리와

크기 회고: changeset 4개(핫픽스 1 + 계획된 3)로 닫혔다 — **milestone 규모가 맞았다**(인플레·디플레
아님). 계획이 예상한 changeset 3 에 예상 못 한 핫픽스 1 이 더해졌다.

- 실표면: 배포 사본(`~/projects/custom-mcps/law-mcp`)을 `git pull && npm run build` 한 뒤 그
  dist 로 `get_law_article` 을 직접 호출해 민법 제245조·근로기준법 제37조·행정심판법 제27조가
  **전부 옳은 법의 옳은 조문**으로 오는 것을 관측했다(수리 전에는 셋 다 다른 법이었다).

  회귀 범위에 넣어야 한다.
