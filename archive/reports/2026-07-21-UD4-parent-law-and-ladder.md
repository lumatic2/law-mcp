# 최종 보고서 — UD4 본법 승격 + 사다리 단축

> 완료: 2026-07-21 · 대상: UD4 (horizon `upstream-delivery`) · 작성: 완료 경계(§B3)

## 1. 문제 정의

UD2 를 닫고 나니 horizon 닫는 기준 5개 중 **하나만 미달**이었다 — 법령 recall@3 ≥85%
(현재 80%). 못 맞히는 5건을 열어 보니 그중 **2건이 같은 결함**이었다: 검색이 시행령·시행규칙은
찾아 놓고 **근거 본법을 안 준다.** 답을 반쯤 찾고 멈춘 상태다.

여기에 UD2 가 남긴 지연 꼬리(F19)를 함께 처리했다. 두 작업의 공통점은 이 horizon 의 축 그대로다 —
**upstream 이 이미 준 것을 우리가 안 쓰고 있다.**

사용자가 "재시작이 귀찮으니 지금 할 수 있는 작업부터"를 지시해, **응답 스키마·도구
description 을 건드리지 않는** 범위로 설계했다.

## 2. Objective 연결

**정확도** 축이 주다. recall@1 72% → 80% 는 소비 LLM 이 목록을 고를 일이 또 줄었다는 뜻이다.
**함정 없음** 축도 밀었다 — 배포 스모크에서 "없는 법령명을 물으면 다른 법 조문이 조용히 오던"
기존 결함을 잡아 경고를 달았다.

## 3. 경로

horizon `upstream-delivery` 의 네 번째 milestone. UD1·UD2·UD3 를 닫은 뒤 남은 갭 하나를
메우려고 새로 개설했다. 계획한 3 step 을 순서대로 밟았다.

계획과 다른 점: step-2 의 **진단이 틀렸다**(아래 이슈 2).

## 4. 구현 결과

**검색이 답을 끝까지 준다.** dev recall@3 80% → **88.0%**(×3, σ=0), recall@1 72% → **80.0%**.
노동·민사·세무가 100% 가 됐다.

```
매입세액 불공제 대상
  전: 부가가치세법 시행령 / 조세특례제한법 / 부가가치세법 시행규칙
  후: 부가가치세법 / 조세특례제한법 / 부가가치세법 시행규칙
```

**검색이 빨라졌다.** 1회 중앙 2939ms → 1966ms.

**horizon 닫는 기준 5개가 모두 충족됐다.** 미달이던 ②를 이 milestone 이 메웠다.

## 5. 이슈와 해결

**이슈 1 — 끼워 넣기가 정답을 밀어냈다(내가 만든 결함, A/B 가 적발).**
첫 구현은 본법을 맨 앞에 끼워 넣었다. 이득 2 / **손실 1** — `사용자책임 면책 사유` 의 정답
민법이 3위에서 잘려 나갔다. 목록은 `limit` 에서 잘리므로 앞에 하나 넣으면 꼬리가 떨어진다.
그 하위법령의 **자리를 본법이 차지**하게 바꾸니 길이가 그대로라 아무도 안 밀렸다(이득 2 / 손실 0).
**집계로는 둘 다 개선이다**(84% vs 88%) — 쿼리 단위 승패가 아니었으면 회귀를 얹은 채 채택했다.

**이슈 2 — 계획의 진단이 틀렸다.**
계획은 "`aiSearch` 가 답했으면 사다리 하위 단계 생략"이었는데, 코드를 열어 보니 **사다리는 이미
첫 단에서 결과가 나오면 멈춘다** — 생략할 게 없었다. 실제 꼬리는 **용어 연계가 사다리 뒤에
순차로 붙어 있던 것**이고, UD2 가 `aiSearch` 에 대해 고친 것과 같은 구조 결함이었다.
같은 처방(동시 실행)으로 해결했고, 캐시만 미리 채우므로 **품질 중립이 설계로 보장된다.**

**이슈 3 — 배포 스모크가 기존 결함을 하나 더 잡았다(near-miss).**
`민법 시행령`(실재하지 않는 법령)을 물으면 **난민법 시행령** 조문이 조용히 왔다 — 문자열 포함
우연이다. UD4 회귀가 아니라 원래 있던 결함이고 UD0·LB3 `ordin` 과 같은 부류다. 마지막 폴백은
`부가가치세` → `부가가치세법` 같은 조회를 살리는 장치라 없애지 않고, **어느 법으로 해석했는지
경고로 남겼다.** 조용한 오답을 시끄러운 오답으로 바꿨다.

**이슈 4 — 지연 최대값은 판정할 수 없었다.**
DoD 는 "중앙·최대 모두 감소"였는데, `지상권 존속기간` 이 3956~9799ms 로 널뛰고 **승격을 끈**
상태가 오히려 9799ms 였다. upstream 변동이 지배하는 구간이라 개선/악화를 말할 수 없다.
**미달이 아니라 측정 불가**로 기록했다.

## 6. 결과물과 증거

changeset 3개: `20260721-ud4-parent-law` · `-ladder-shortcut` · `-verdict`.
커밋: `49b3304` · `f41cf5f` · `2cb089d`.
증거: `evidence/bench/2026-07-21-ud4-verdict.md`.

검증: `npm test` **175/175**(169 → 175, 신설 18건). 교차 A/B 2회 결과 동일.
`bench:golden --repeat 3` 88.0% ×3.

실패 경로도 실제로 실행했다: 실재하지 않는 본법명이 복원되는 입력에서 **편입하지 않는 것**을
확인했고, 승격 단계가 던져도 검색이 죽지 않는지 확인하다 **실패 흡수가 조회 함수 안에만 있던
결함을 테스트가 적발**해 단계 전체를 감쌌으며, 프리페치를 HTTP 503 으로 죽여도 검색이 정상인
것을 관측했다.

크기 회고: changeset 3개로 닫혔다 — **milestone 규모가 맞았다**(계획한 3개와 정확히 일치).

- 실표면: 배포 사본을 `git pull && npm run build` 한 뒤 그 dist 로 `search_law("매입세액 불공제
  대상")` → **부가가치세법** 1위, `search_law("사용자책임 면책 사유")` → 민법 3위 유지,
  `get_law_article("민법","제1조")` → 정상(승격 미오염)을 관측했다. 이 스모크가 이슈 3 을 잡았다.
- 재현: `npm test && npm run bench:golden -- --set golden-v2 --split dev --repeat 3`

## 7. 후속 제안

- **horizon close 판단이 다음이다** — 닫는 기준 5개가 **모두 충족**됐다. close 는 홀드아웃
  15건을 1회 개봉한다는 뜻이고(코드로 1회만 허용), 열면 그 세트는 죽는다. 사용자 판단 사항.
- **일반성은 아직 실측이 아니다** — 본법 승격 규칙에 법명 하드코딩이 없다는 것은 설계 논증이지
  홀드아웃 검증이 아니다. close 시 개봉하면 그 자리에서 판명된다.
- **행정 60%** 가 유일한 약점 도메인이다. 못 맞히는 3건(처분 사전통지 의무 · 과징금 부과 기준 ·
  범죄수익 몰수 추징)은 상류가 못 주는 것이라 랭킹으로 안 풀린다.
- **F6**: `aiRltLs`·`lsRlt` 는 본법-하위법령 관계를 upstream 이 직접 주는 경로일 수 있다.
  이번엔 이름 복원으로 했고, 그 API 가 더 정확하면 갈아탈 후보다.
- **F20**(신규): 이름 해석 폴백이 부분문자열 우연으로 다른 법을 집는다. 이번엔 경고로 덮었지만
  근본 해결(질의어가 결과 이름의 부분문자열인지 검사)은 남아 있다.
- **MCP 재시작은 이 milestone 때문에는 불요** — UD2·UD3 의 표면 변경 부채는 그대로다.
