# AR4 완료 + horizon `agentic-reach` 종료

> milestone: AR4 · horizon: agentic-reach · 2026-07-23

## 1. 결과

**horizon 을 닫는다.** 닫는 기준 6종 전부 충족.

| # | 기준 | 실측 | 판정 |
|---|---|---|---|
| 1 | 채점에 LLM judge 0개 | 채점기 LLM 호출 0 · 결정성 프로브 통과 · selftest 9/9 | 충족 |
| 2 | 반복 없는 수치 출하 금지 | 리포터가 반복 1회 입력을 기계 거부 | 충족 |
| 3 | 맥락 유출 0건 | 34건 0 · 유출 주입 시 6건 적발 + exit 1 | 충족 |
| 4 | 단발 vs 에이전트형 대조 | **75% → 90%** (`pass^3`) | 충족 |
| 5 | 기권 정밀도·재현율 쌍 | **100% / 100%** | 충족 |
| 6 | 범용 회귀 ≥88% | **88.0%** | 충족 |

**이 horizon 이 만든 것은 도구 개선이 아니라 자다.** 같은 20건에서 단발 recall@3 75% 가
에이전트형 `pass^3` 90% 로 읽혔다 — 도구는 그대로다. tax-vertical 에서 미달로 기록한
"세법 recall@3 ≥90%"는 도구가 못 한 게 아니라 **단발로 재서 못 한 것으로 보인** 것이었다.

`SR@t` 가 기전이다: 첫 질의 80% → 재질의 한 번에 95%. 그 **15%p 가 단발이 버리던 값**이고,
t3 부터 안 오르는 것은 **루프가 만능이 아니라 한 번의 재질의가 거의 전부**라는 뜻이다.

### ⚠ 홀드아웃 수치는 신뢰하지 않는다 (사용자 확정 2026-07-23)

홀드아웃 10건 × 5회 = `pass^5` **100%** · `SR@1` 98%. dev 보다 높으므로 **과적합은 기각**된다.
**그러나 절대 수치는 채택하지 않는다.**

케이스 출처를 대조하면 dev 는 **20/20 이 구 세트**(단발 벤치용으로 독립 작성)인데
홀드아웃은 **6/10 이 에이전트가 주제까지 고른 것**이고, 고를 때 도구로 조문 존재를 확인했다.
**도구가 잘 다루는 영역에서 문제를 낸 것**이다. 유출 탐지기는 문자열만 보므로 이런
**구성에 의한 오염**을 못 잡는다 — 프리모템 ①이 문자열이 아니라 **주제 선택 경로**로 발화했다.

**결론: 과적합 기각만 살리고 100% 라는 수치는 근거로 쓰지 않는다.**
다음 horizon 은 **홀드아웃 주제를 에이전트가 고르지 않는 절차**부터 만들어야 한다.

## 2. 이슈와 해결

**① 계획이 API 키를 요구해 막혔다.** 평가 에이전트를 SDK 로 새로 호출하는 설계였다.
사용자 지적("너가 LLM인데 왜 새 LLM이 필요하지")으로 **살아있는 세션이 곧 소비자**임을 확인,
SDK 경로를 폐기했다. 추정 비용 $1~3/완주 → **0**. 프리모템 ⑤(비용 폭발)가 이 전환으로 소멸했다.

**② 블라인드 arm 이 3회 막혔다.** Codex 서브에이전트가 `npx` 실패 → `tsx` EPERM →
**네트워크 차단**까지 계단식. 근본 원인은 샌드박스 오프라인. `orca terminal create` 로
**실제 세션**을 띄워 해결했다. 1차 시도에서 **도구가 죽었는데 답을 지어낸 것**을 발견해
드라이버에 `tool_ok: false` 실행 채점 거부를 박았다 — 실행 실패가 도구 성능으로 둔갑하는 것이
이 horizon 이 막으려던 왜곡의 전형이다.

**③ 내 진단이 틀렸다.** "법령은 맞히고 조문을 놓친다"로 보고 수리를 설계했으나 실측에서
`lawRank=-1` 이 5건 중 4건 — **정답 법령 자체가 후보에 없었다.** 준비한 수리는 기여도 0/5,
**미채택**. 수리 대상을 "일상어 질의의 법령 도달"로 재지정했다.

**④ AR3 는 선언 문턱을 못 넘었다.** 어휘 공백 경고를 "손실 0 이 구조적"만으로 채택하고
`SR@1` 순 이득은 미실측으로 남겼는데, 대조군 A/B 로 갚으니 **`SR@1` ON 80% vs OFF 88.3%
(−8.3%p)** 로 **미달**이었다. 경고 정밀도 75% 라 이미 성공한 검색 넷 중 하나에 재질의를 시킨다.
동시에 `pass@3` 는 100% vs 95% 로 올랐다 — **도달 가능성은 올리고 첫 질의 성공은 내린다.**
어느 쪽이 나은지는 제품 판단이라 **미결 결정으로 남기고 현행(경고 ON)을 유지**했다.

**⑤ 자의 눈금 오류 3건을 스스로 적발했다.** ⓐ 마찰 축이 총 턴이라 `SR@1` 이 구조적으로 0%
(본문 확인을 요구했으므로 1턴 불가) → 검색 횟수 축으로. ⓑ 옳은 기권이 정답률을 깎아
`pass@3` 상한이 83.3% 로 묶임 → 정답률 축에서 분리 + 회귀 프로브. ⓒ 대조군이 `queries: []` 를
보고했는데 0턴으로 받아 `SR@1` 을 부풀림 → 하한 1 로. 셋 다 **자를 만드는 중 자에서 나온
오류**라 그대로 기록했다.

**⑥ 측정 못 한 구간은 못 했다고 남겼다.** 블라인드 arm 이 막혔을 때 추정으로 채우지 않고
`stop_reason=blocked` 로 정지했다.
