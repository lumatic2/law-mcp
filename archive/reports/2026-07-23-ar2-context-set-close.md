# AR2 완료 — 맥락 세트 + 기준선

> milestone: AR2 (horizon `agentic-reach`) · 2026-07-23

## 1. 결과

**잴 것을 만들고 기준선을 박았다.**

세트: dev 24건(구 홀드아웃 20건에 맥락 부착 + 기권 4건) · holdout 10건(신규·봉인).
유출 **0**, 홀드아웃 라벨 10건 전부 실 API 조회로 존재 확인.

기준선 — 정답 케이스 20건 × **블라인드 3회**:

| 지표 | 실측 |
|---|---|
| `pass^3` (3회 전부) | **90.0%** |
| `pass@3` (한 번이라도) | 100.0% |
| `SR@1` (첫 질의) | **80.0%** |
| `SR@t` | 80% → **95%** (t2), 이후 불변 |
| `AT` | 1.16 |
| 기권 정밀도 / 재현율 | **100% / 100%** |
| 침묵률 | 0.0% |

**같은 20건: 단발 recall@3 75% → 에이전트형 `pass^3` 90%.** 도구는 그대로고 재는 방식만
바꿨다. tax-vertical 에서 미달로 기록한 "recall@3 ≥90%"가 여기서 나온다 — 도구가 못 했던 게
아니라 단발로 재서 못 한 것으로 보였다. **이 horizon 의 명제가 확정 수치로 지지됐다.**

`SR@t` 가 메커니즘이다: 첫 질의 80% → 재질의 한 번에 95%. 그 **15%p 가 단발이 버리던 값**이고,
t3 부터 안 오르는 것은 **루프가 만능이 아니라 한 번의 재질의가 거의 전부**라는 뜻이다.

## 2. 이슈와 해결

**① 블라인드 arm 이 3회 막혔다.** Codex 서브에이전트로 재려 했으나 `npx` 실패 → `tsx` EPERM →
**네트워크 차단**(`connect EACCES :443`)까지 계단식으로 막혔다. 근본 원인은 샌드박스가 오프라인.
사용자 지시로 `orca terminal create` 로 **실제 세션 3개**를 띄워 해결했다.

**② 1차 시도가 이 벤치의 존재 이유를 증명했다.** 도구가 전부 죽었는데 Codex 는 3건 모두
`not_found` 로 답했다. 그대로 받았으면 "에이전트가 못 찾았다"가 기준선이 됐을 것이다.
드라이버에 **`tool_ok: false` 실행은 채점 거부**를 박았다.

**③ 내 자의 눈금 오류 2건을 스스로 적발했다.**
- 마찰 축이 총 턴이라 `SR@1` 이 구조적으로 0% 였다("지목 전 본문 확인"을 요구했으므로 1턴 불가).
  → 검색 횟수 축으로 교체.
- **옳은 기권이 정답률을 깎고 있었다** — `pass@3` 상한이 20/24=83.3% 로 묶였다.
  → 정답률 축에서 기권 분리 + 회귀 프로브(리포터 selftest ⑥).

**④ 측정을 못 한 구간은 못 했다고 남겼다.** AR3 의 `SR@1` 순 이득은 수리 후 arm 만 있어
여전히 미실측이다. 추정으로 채우지 않았다.
