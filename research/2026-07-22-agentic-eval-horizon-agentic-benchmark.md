# 에이전트형 평가 설계 리서치 — turns-to-answer / 정직한 실패율

> 소비처: law-mcp 다음 horizon 설계(에이전트형 평가)
> 작성일: 2026-07-22 (모든 인용 접근일 동일)

## 0. 문제 정의 재확인

지금 벤치는 라벨 쿼리 1개 → `search_law` 1콜 → recall@3 채점, 단발이다. 실측(레포 내부)으로 확인된 것: 맥락 없는 원 쿼리는 실패하고, 대화 맥락을 얹은 재질의는 성공한다. 이건 "제품이 실패했다"가 아니라 "벤치가 재질의 루프를 측정 안 한다"는 것이므로, 다음 horizon 은 평가 자체를 멀티턴·에이전트형으로 바꿔야 한다는 게 출발점이다. 동시에 이 레포의 자산은 측정 규율(교차측정으로 회귀 오독을 잡은 전례)이므로, 새 지표가 비결정성 때문에 그 규율을 깨면 안 된다.

## 1. 선행 사례가 비결정성을 다룬 방법

### 1-1. τ-bench / τ²-bench (Sierra Research)

- τ-bench 는 에이전트·사용자 시뮬레이터(둘 다 LLM)·정책 문서·DB(로컬 JSON) 로 구성된 고객지원 시뮬레이션이다. 대화가 끝나면 **DB 최종 상태를 주석 처리된 목표 상태와 비교**해 채점한다 — 자연어 출력 자체를 LLM judge 로 채점하지 않고 상태 비교로 판정하는 게 결정성 확보의 핵심 장치다.
  출처: https://arxiv.org/abs/2406.12045 (접근 2026-07-22)
- 반복 실행 자체는 여전히 비결정적(사용자 시뮬레이터도 LLM, 에이전트도 LLM)이라는 걸 τ-bench 저자들이 정면으로 인정하고, 그래서 **pass^k** 라는 새 지표를 만들었다: 같은 과제를 k번 독립 시행해서 "k번 다 성공"한 비율을 잰다. pass@k(적어도 1번 성공)와 반대 방향 — "성공률이 아니라 성공의 일관성"을 잰다. GPT-4o가 retail 도메인에서 pass^1 은 높아도 pass^8 < 25%로 급락하는 사례가 보고됨.
  출처: https://huggingface.co/papers/2406.12045 (접근 2026-07-22)
- τ²-bench(2025)는 dual-control 확장(에이전트·사용자 둘 다 tool call 가능)이라 이 레포와는 직접 관련 낮음. 저장소 README 는 `--num-trials` 옵션으로 반복 횟수를 CLI 인자화하고, 버전 간 평가 기준이 바뀌면 `pre-v1.0.1` 같은 태그로 과거 동작을 재현하도록 안내한다 — "평가 자체도 버저닝 대상"이라는 태도.
  출처: https://github.com/sierra-research/tau2-bench (접근 2026-07-22)
- 확인 못 함: 논문 PDF 원문에서 pass^k 의 정확한 수식과 실험에 쓴 k 값 목록, temperature 세팅 — WebFetch 로 PDF 텍스트 추출이 실패해 이차 출처(HuggingFace papers 페이지)로만 확인.

### 1-2. SWE-bench

- 평가 하네스는 **Docker 완전 컨테이너화**로 환경 재현성을 확보한다(코드 실행 결과의 결정성 확보 — "테스트 통과/실패"라는 이진 채점 자체는 결정적).
  출처: https://www.swebench.com/SWE-bench/ (접근 2026-07-22)
- release 1.x 까지는 temperature=0 고정을 권장했으나 release 2.x 부터는 temperature 파라미터 자체를 강제하지 않는 쪽으로 바뀜 — 즉 "temperature=0 이면 재현된다"는 가정을 실무적으로 포기하는 추세.
- 별개 연구에 따르면 temperature=0·seed 고정 상태에서도 **원문 출력(raw output) 자체의 재현성은 최소 수준**이며, 모델·과제에 따라 accuracy 편차가 작을 수도 클 수도 있다는 게 실측됨 — "결정론적 세팅 = 결정론적 결과"가 실제로는 성립하지 않는다는 경고.
  출처: https://arxiv.org/pdf/2604.21480 (Efficient Agent Evaluation via Diversity-Guided User Simulation, 접근 2026-07-22)

### 1-3. BFCL (Berkeley Function Calling Leaderboard)

- 이 레포 상황과 가장 가까운 참조점: **AST(추상 구문 트리) 매칭**으로 "함수를 실제로 실행하지 않고" 호출의 구조(함수명·인자)만 정답과 비교한다. LLM judge 를 안 쓰기 때문에 "네 개 벤치마크 중 가장 재현성이 높다"는 평가를 받음.
  출처: https://gorilla.cs.berkeley.edu/leaderboard.html , https://proceedings.mlr.press/v267/patil25a.html (접근 2026-07-22)
- 재현 절차는 특정 커밋 체크아웃 또는 `pip install bfcl-eval==<날짜버전>` 로 고정 — "코드/데이터셋 버전 고정"이 재현성 확보의 실무적 축.
- 이 접근을 law-mcp 에 옮기면: "에이전트가 최종적으로 도달한 조문 ID가 정답 조문 ID와 일치하는가"처럼 **자연어 출력이 아니라 구조화된 최종 상태(도구 호출 인자·반환된 법령ID)를 비교**하는 채점기를 만들 수 있다는 뜻. LLM judge 없이도 "정답 조문 도달 여부"는 이미 구조화 값이라 이 방식이 잘 맞는다.

### 1-4. WebArena → WebArena Verified

- 원조 WebArena 는 단일 실행 평균 성공률만 보고했는데, 후속 검증판(WebArena Verified)이 지적한 문제: **같은 태스크셋에서 두 번 실행한 집계 성공률은 거의 같아도, 개별 태스크 단위로는 상당수가 뒤집힌다** — 집계 안정성이 태스크 단위 노이즈를 가린다는 것. (이 레포가 오늘 겪은 "단일 측정 오독" 경험과 정확히 같은 패턴.)
- 개선판은 812개 태스크 전수 감사, 모호한 지시문 정리, 그리고 보고 방식 자체를 바꿔 **template-level macro average + 95% 신뢰구간 + 실패유형 분해**를 표준으로 채택.
  출처: https://openreview.net/forum?id=94tlGxmqkN (WebArena Verified, 접근 2026-07-22) / 요약 인용: WebSearch 결과, 원문 링크 상동

### 1-5. MCP 전용 평가 (2026년 신규)

- **MCP-Bench**(ICLR 2026): 28개 실서비스 MCP 서버·250개 도구를 연결해 104개 과제를 생성, 규칙 기반 채점 + LLM-as-judge 혼합. "표준화된 multi-pass 평가"(같은 판정을 여러 번 반복해 평균)로 judge 분산을 줄인다고 보고.
  출처: https://openreview.net/pdf?id=fe8mzHwMxN (접근 2026-07-22)
- **MCPMark**: Notion/GitHub/Filesystem/Postgres/Playwright 등 127개 과제, 최고 모델(gpt-5-medium)도 pass@1 52.56%에 그침 — MCP 실사용 과제의 난도가 매우 높다는 근거.
  출처: https://openreview.net/forum?id=uobROwBsJm (접근 2026-07-22)
- **경고 신호 — "Benchmarking the Benchmarks: A Validity Audit of Tool-Calling Evaluation"**: LiveMCPBench 를 23회 반복 실행했더니 점수가 57.9%~76.8%로 **18.9%p 스프레드**가 나왔고, 이는 리더보드 순위를 뒤집을 만한 크기다. 원인으로 "확률적 채점 기준(stochastic rubric)"과 "judge instability"를 지목. τ²-bench 계열 벤치도 "평가 대상이 뭔지 불명확하게 전달되는" 문제(모델에게 어떤 도구를 호출해야 하는지가 태스크 설계상 모호함)를 지적당함.
  출처: https://arxiv.org/pdf/2607.02577 (접근 2026-07-22)
  → **이 레포에 가장 직접적인 경고**: LLM-as-judge 를 채점 파이프라인에 넣는 순간, 반복 실행 자체가 표준 관행이 되어야 하고, 안 하면 스코어가 노이즈인지 실력차인지 구분이 안 된다.

## 2. pass@k / pass^k 계열 신뢰도 지표

- **pass@k** (Codex 논문, OpenAI, "Evaluating Large Language Models Trained on Code", arXiv:2107.03374): n개 샘플 생성, c개 정답일 때 unbiased estimator는
  `pass@k = 1 - C(n-c, k) / C(n, k)`
  (naive하게 `1-(1-c/n)^k` 로 계산하면 복원추출을 가정하는 셈이라 과소추정됨 — 실제는 비복원추출이므로 조합론적 U-statistic 보정이 필요).
  출처: https://arxiv.org/pdf/2107.03374 , 해설: https://leehanchung.github.io/blogs/2025/09/08/pass-at-k/ (접근 2026-07-22)
- **pass^k**(τ-bench): "k번 중 적어도 1번"이 아니라 "**k번 다** 성공"을 재는 지표. 수학적으로 단일 성공확률 p 에 대해 `pass^k ≈ p^k` 로 지수 감쇠한다 — 90% 성공률(pass@1)도 k=8이면 57%로 떨어진다는 예시가 보고됨.
  출처: https://huggingface.co/papers/2406.12045 (접근 2026-07-22)
- **몇 회가 실무적으로 충분한가**: 명시적 "표준 k값"은 확인 못 함. 관찰된 실무: τ-bench 는 k=1,2,4,8 등 여러 k 를 동시에 보고(정확한 목록은 원문 PDF 파싱 실패로 미확인), WebArena Verified 는 반복 실행 + 95% CI 로 대체, "Validity Audit" 논문은 최소 20회 이상 반복(LiveMCPBench 사례 23회)해야 변동폭이 드러난다는 실측을 제시. 즉 **"k=1은 항상 부족", "3~5회면 방향은 보이나 신뢰구간은 넓다", "20회 이상이라야 변동폭 자체를 정량화할 수 있다"**는 게 종합적 시사점이지 고정된 표준 횟수는 없다.
- 이 레포 규모(케이스 수 적음, LLM 호출 비용 있음)에 옮기면: **pass@3**(3회 중 1회라도 도달=현재 채점과 유사한 관대한 지표) 과 **pass^3**(3회 다 도달=일관성 지표)을 동시에 보고하는 절충안이 τ-bench 철학과 맞고 비용도 감당 가능한 수준.

## 3. 비용 통제 기법과 함정

- **prompt caching**: 반복되는 프리픽스(시스템 프롬프트·도구 스키마·정책 문서)를 캐싱해 API 비용 41~80% 절감, TTFT 13~31% 개선이 보고됨. 단, "Don't Break the Cache" 연구는 장기 horizon 에이전트 태스크에서 캐시가 깨지는(cache miss) 조건을 별도로 분석 — 프리픽스가 조금만 바뀌어도 캐시 이득이 사라진다는 함정.
  출처: https://arxiv.org/pdf/2601.06007 (접근 2026-07-22)
- **plan caching / record & replay**: 에이전트가 이미 만든 실행 계획이나 도구 호출 시퀀스를 기록해두고, 같은 패턴이 재등장하면 LLM 호출 자체를 스킵하고 재생(replay)한다. GAIA 등에서 비용 50% 절감, 지연 27% 절감, 성능은 96.6% 유지라는 보고가 있음.
  출처: https://arxiv.org/abs/2506.14852 , https://arxiv.org/html/2505.17716v1 (접근 2026-07-22)
  **함정**: 이 기법은 "같은 입력엔 같은 계획"을 전제하므로, 벤치마크 채점용으로 쓰면 오히려 "이번에 진짜 재추론했는지"를 가려버릴 위험이 있다 — 비용 절감 장치를 평가 루프에 그대로 꽂으면 측정하려는 대상(에이전트의 실제 재질의 능력)을 캐시가 대신 answer 해버릴 수 있다.
- **소형 모델 대체·샘플링**: 명시적 사례는 확인 못 함(이번 리서치 범위에서 law-mcp 벤치에 해당하는 사례를 찾지 못함). 다만 일반 원칙으로 "user simulator 역할은 저비용 모델, 실제 평가 대상 에이전트는 프로덕션 모델"로 비대칭 배치하는 것이 τ-bench 류 설계의 관례로 보이나, 이 배치 자체를 정량 검증한 논문은 확인 못 함 — 추정 표기로 남김.

## 4. 사용자 시뮬레이터의 실패 모드와 완화책

- **leakage(정답 흘리기)**: 시뮬레이터가 원래 의도(gold intent)를 알고 있으므로, 대화 중 무심코 정답 키워드·식별자를 흘리면 에이전트의 실제 검색 능력이 아니라 "받아쓰기 능력"을 재게 된다. 완화책으로 보고된 것:
  - **명시적 금지 지시**: 시뮬레이터 프롬프트에 "정답 논문 제목/저자명 언급 금지" 등을 박아넣기.
  - **결정적 문자열 매칭 탐지기**: 시뮬레이터 발화에 gold 식별자의 부분 문자열이 등장하는지 사후 검사. 이 완화책 적용 후 매치율이 24.2%→45.3%로 바뀌었다는(leakage 가 실제로 결과를 왜곡하고 있었다는) 실측 보고.
  - **prompt vaguenization**(UserVille): 애초에 벤치마크 프롬프트를 정밀한 것에서 의도적으로 모호한 변형으로 바꿔, 에이전트는 모호한 것만 보고 시뮬레이터만 원문 의도를 쥐고 있게 설계 — 정보 비대칭을 구조적으로 만든다.
  출처: https://futureagi.com/glossary/multi-turn-llm-conversation-degradation/ , https://arxiv.org/html/2605.20204 (RealUserSim, 접근 2026-07-22)
- **시뮬레이터가 실제 사용자의 대리인으로 타당한가**: SimulatorArena(909개 사람-LLM 대화 주석 벤치)는 "사용자 프로필로 조건화된 시뮬레이터"가 인간 평가와 스피어만 상관 0.7 정도에 도달한다고 보고 — 완벽하지 않지만 실용적 대안 수준. 구체적 실패 모드(과도한 협조성 등)는 초록 수준에서는 명시되지 않아 **확인 못 함**(원문 PDF 상세 미확인).
  출처: https://arxiv.org/abs/2510.05444 (접근 2026-07-22)
- law-mcp 맥락 적용: "맥락 한 단어를 얹으면 성공한다"는 이 레포의 실측 사례 자체가, 시뮬레이터가 얹는 맥락의 "양"을 통제해야 한다는 뜻이다 — 시뮬레이터가 처음부터 정답급 맥락(법령명 근접어 등)을 흘리면 벤치가 다시 낙관적으로 왜곡된다. 시뮬레이터 프롬프트에 "질의자는 법률 비전문가이며 법령명·조문번호를 모른다"는 페르소나 제약을 명시하는 게 최소 방어선.

## 5. turns-to-answer 류 지표의 선행 사례

- 직접 이 이름 그대로 쓰는 유명 벤치는 확인 못 함. 대신 근접 개념들이 존재:
  - **SR@t (Success Rate at turn t)**: 대화형 추천 시스템 분야에서 "t번째 턴까지 몇 %가 성공했는가"의 누적 곡선. **AT(Average Turns)**: 성공한 대화들의 평균 턴 수 — 이게 사실상 "turns-to-answer"의 선행 명칭.
    출처: 검색결과(Knowledge Graph-enhanced Sampling for Conversational Recommender System 등 대화형 추천 문헌, 접근 2026-07-22)
  - **Turn Success Rate / Averaged Turn Success Rate(ATS)**: 턴 단위 이진 성공(0/1)을 평균 — "몇 턴에 도달했나"보다는 "각 턴이 정답이었나"에 가까움, law-mcp 목적(최종 도달까지 걸린 턴)과는 결이 조금 다름.
  - 복합 확률 경고: "턴당 정확도 95%도 5턴이면 누적 성공률 77%로 떨어진다"는 예시가 여러 문헌에서 반복 인용됨 — turns-to-answer 를 채택하면 "짧을수록 좋다"는 것 자체가 턴당 실패 확률의 복리 효과를 반영한다는 근거.
  출처: https://futureagi.com/blog/stress-test-llm-2025/ , 대화형 추천시스템 문헌(접근 2026-07-22)
- 결론: "turns-to-answer" 라는 정확한 명칭의 표준 벤치는 확인 못 함. 가장 가까운 정본 명칭은 **AT(Average Turns to success)** + **SR@t 곡선**의 조합이며, law-mcp 에 옮길 이름은 이 둘 중 하나를 그대로 차용하거나 "도달 턴수"로 국문 명명하는 게 안전하다.

## 6. 정직한 실패(abstention) 평가 — selective prediction / coverage-risk

- **이론적 뿌리**: El-Yaniv & Wiener, "On the Foundations of Noise-Free Selective Classification", JMLR 2010. **risk-coverage curve** 개념의 출처 — coverage(답한 비율)를 x축, risk(오답률)를 y축으로 놓고 threshold 를 스윕하며 그리는 곡선. 요약을 위해 **AURC(Area Under Risk-Coverage curve)** 를 단일 스칼라로 보고 — 낮을수록 좋음.
  출처: 검색결과 인용(Semantic Scholar 서지 확인, 원문 JMLR 링크는 이번 조사에서 직접 확인 못함 — 서지사항만 확인, 원문 URL **확인 못 함**)
- **RAG 특화 사례**: "abstain unless evidence supports the answer" 프레임(SURE-RAG 등)은 정답/오답/불충분(insufficient) 3분류 후 불충분이면 기권. Risk 를 `1 - R@K`로 정의하고 threshold 스윕으로 risk-coverage 곡선을 그리는 방식이 검색 결과에서도 그대로 재사용됨.
  출처: https://arxiv.org/html/2605.03534 , https://arxiv.org/html/2605.18792 (접근 2026-07-22)
- **QA 표준 지표**: SQuAD 2.0 이 "답 없음" 질문 5만여 개를 추가해 "모른다고 말할 수 있는가"를 정식 평가 축으로 만든 원조 사례(2018). 최근에는 **AbstentionBench** 프레임의 Abstention F1(정밀도·재현율의 조화평균), Abstention Recall/Precision 을 표준 지표로 인용.
  출처: 검색결과(AbstentionBench 계열, RefusalBench arXiv:2510.10390, 접근 2026-07-22)
- law-mcp 적용: "정직한 실패율"을 SQuAD 2.0/AbstentionBench 식으로 옮기면 — 질의셋에 **의도적으로 답이 없는(현행법에 해당 조문이 없는) 케이스**를 섞어 놓고, 에이전트가 "모른다"고 답하는지(recall), 답이 있는데 괜히 기권하지 않는지(precision)를 같이 재는 게 정본 설계다. 지금 구상처럼 "못 찾을 때 못 찾았다고 말하나"만 재면 recall 쪽만 재는 셈이라 precision(과잉 기권) 쪽도 균형 있게 넣어야 selective prediction 문헌과 정합된다.

## 7. 이 레포에 적용할 권고

### 권고 1 — 채점은 자연어 판정이 아니라 구조화 상태 비교로 한다 (BFCL·τ-bench 패턴 차용)
- **무엇을**: "정답 조문에 도달했는가"의 판정을, LLM judge 로 최종 답변 텍스트를 읽게 하지 말고 **도구 호출 로그의 최종 `get_law_article` 인자(법령ID+조문번호)를 정답 튜플과 문자열/ID 비교**하는 결정적 채점기로 만든다.
- **왜**: BFCL 이 "네 벤치마크 중 가장 재현성 높다"고 평가받는 이유가 바로 이거고, τ-bench 도 자연어가 아니라 DB 최종 상태로 채점한다. law-mcp 는 이미 구조화 도구 반환값(법령ID·조문번호)이 있으니 이 패턴을 그대로 쓸 수 있는 유리한 위치다.
- **알려진 함정**: 도구가 리랭킹·유사조문 여러 개를 반환하면 "정답이 상위 N개 안에 있는가"로 다시 recall@N 문제가 재발한다 — 멀티턴이어도 마지막에 결국 top-N 판정으로 퇴화하지 않도록, "에이전트가 스스로 하나를 확정 지목했는가"까지 요구하는 종료 조건을 태스크 설계에 넣어야 한다.

### 권고 2 — 단일 실행 절대 금지, pass@3 + pass^3 을 동시에 보고한다 (τ-bench 패턴)
- **무엇을**: 케이스당 최소 3회 독립 실행(다른 seed/온도) 후 pass@3(3회 중 1회 도달)과 pass^3(3회 다 도달)을 같이 보고. WebArena Verified 식으로 신뢰구간(또는 최소·최대 범위)도 같이 적는다.
- **왜**: 오늘 이 레포가 recall@1 80→68% 를 회귀로 오독할 뻔했던 사건 자체가 단일측정의 위험을 실증했다. "Validity Audit" 논문의 LiveMCPBench 23회 반복 사례(57.9~76.8%, 18.9%p 스프레드)는 반복 없는 벤치가 순위를 뒤집을 만큼 위험하다는 걸 정량적으로 보여준다.
- **알려진 함정**: 3회는 방향성 확인용이지 통계적으로 충분하지 않다(20회 이상이라야 변동폭이 안정된다는 사례도 있음). 비용 때문에 3회로 타협한다면, "3회는 잠정치이며 horizon 종료 판정처럼 중요한 결정에는 반복 횟수를 늘린다"는 규칙을 명시해야 한다.

### 권고 3 — 사용자 시뮬레이터에 "맥락 누출 금지" 페르소나 제약과 leakage 탐지기를 같이 넣는다
- **무엇을**: 시뮬레이터 프롬프트에 "질의자는 법률 비전문가, 정확한 법령명·조문번호를 모른다"는 페르소나를 명시하고, 시뮬레이터 발화에 정답 법령명/조문번호의 부분 문자열이 등장하면 그 turn 을 자동 플래그하는 사후 탐지기를 채점 파이프라인에 넣는다.
- **왜**: leakage 완화 전/후 매치율이 24.2%→45.3%로 바뀐 실측 사례가 "완화 안 하면 벤치가 낙관적으로 왜곡된다"를 보여준다. law-mcp 실측(맥락 한 단어로 실패→성공)이 정확히 이 현상의 사촌이라, 시뮬레이터가 그 "한 단어"를 얼마나 후하게 주는지가 지표를 크게 흔든다.
- **알려진 함정**: 과도하게 제약하면 시뮬레이터가 비현실적으로 무능한 사용자가 되어(실제 사용자보다 덜 협조적) 벤치가 이번엔 비관적으로 왜곡된다. SimulatorArena 가 인간 대화 상관 0.7 수준에서 그친다는 것도 "시뮬레이터=완벽한 사람 대리인"은 아니라는 뜻이므로, 페르소나 제약 수위는 실제 사용자 대화 로그(있다면)로 캘리브레이션해야 한다.

### 권고 4 — turns-to-answer 는 "AT(평균 도달 턴수) + SR@t 곡선"으로 구체화한다
- **무엇을**: 단일 숫자(turns-to-answer 평균)만 보고하지 말고, 대화형 추천 문헌의 SR@t 처럼 "1턴째 성공률, 2턴째 누적 성공률, 3턴째…" 곡선을 같이 보고한다.
- **왜**: 정확히 이 이름의 표준 벤치는 확인 못 했지만, 근접 지표(AT/SR@t)가 이미 존재하고, 이 레포의 실측(1턴 실패→2턴 성공)이 바로 이 곡선의 첫 두 점이다. 단일 평균은 "1턴만에 늘 성공하는 에이전트"와 "가끔 5턴 걸리지만 결국 성공하는 에이전트"를 구분 못 한다.
- **알려진 함정**: 턴 수 상한(cutoff)을 어디로 잡느냐에 따라 곡선 모양이 크게 바뀐다 — cutoff 를 너무 낮게 잡으면 "정직한 실패"와 "포기 안 하면 됐을 실패"가 뒤섞인다. cutoff 는 별도 실험(사용자가 실제로 몇 턴까지 참는지)으로 정하거나, 최소한 명시적으로 고정하고 문서화해야 한다.

### 권고 5 — "정직한 실패율"은 recall 뿐 아니라 precision 도 같이 잰다 (selective prediction 패턴)
- **무엇을**: 답이 없는(현행 법령에 해당 조문이 없거나 폐지된) 케이스를 질의셋에 섞어 두고, ① 답 없을 때 "모른다"고 답하는 비율(Abstention Recall) ② 답 있는데 괜히 "모른다"고 하지 않는 비율(Abstention Precision, 또는 1-과잉기권율)을 같이 보고한다. 가능하면 threshold 스윕으로 risk-coverage 곡선/AURC 까지.
- **왜**: SQuAD 2.0 이래 "정직한 실패"는 항상 recall-precision 쌍으로 재는 게 정본이다. recall 만 재면 "항상 모른다고 답하는 에이전트"가 만점을 받는 퇴화 사례를 못 거른다.
- **알려진 함정**: 법률 도메인에서 "답이 없는 케이스"를 인위적으로 만들기 어렵다(실제로 존재하지 않는 조문을 묻는 질의를 자연스럽게 설계해야 함) — 너무 티나게 만든 함정 질의는 실제 사용자 분포를 대표하지 못해 precision 지표 자체가 왜곡될 수 있다.
