# 리서치 — 법제처 DRF OPEN API 전수 조사

- 일자: 2026-07-21 · 소비처: 다음 horizon/milestone 설계
- 방법: ① **실 API 전수 프로브**(타깃 후보 83종, 3차에 걸쳐) ② **공식 가이드 목록 조사**
  (`https://open.law.go.kr/LSO/openApi/guideList.do`, 접근 2026-07-21 — **총 191건** 표기).
  출처 = 응답 자체 + 공식 가이드 페이지.

## 왜 지금 이 조사인가

이 레포는 같은 실수를 **두 번** 했다. LB3 에서 "법원 5종이 실재하는데 미지원"을 뒤늦게 발견했고,
LB5 에서 "용어 연계 색인이 이미 있는데 미사용"을 또 발견했다. 둘 다 착수 후에 알았다.
전수 조사를 먼저 했다면 계획의 순서가 달라졌을 것이다. **이 문서가 그 부채를 갚는다.**

## 요약 — 우리가 쓰는 것 vs 있는 것

공식 가이드가 스스로 **191건**을 나열한다. 우리가 쓰는 것은 **8종**이다.

| 구분 | 개수 | 목록 |
|---|---|---|
| **현재 사용 중** | 8 | law, prec, admrul, expc, detc, decc, ordin, lstrm(+lstrmAI/lstrmRltJo) |
| **실 API 로 실재 확인·미사용** | **25+** | 아래 |
| 공식 문서에만 있고 이 OC 로 미확인 | 다수 | 중앙부처 39종 해석·특별행정심판 4종·맞춤형 6종 등 |

## ★ 최대 발견 — `aiSearch` (법제처 자체 지능형 검색)이 우리보다 낫다

법제처는 **자연어 질의를 조문 단위로 답하는 검색**을 이미 제공한다. 같은 dev 골든셋 25건으로
우리 파이프라인과 직접 비교했다.

| 지표 | 우리 (LB5 완료, dev 튜닝됨) | **`aiSearch` 단독 (무튜닝)** |
|---|---|---|
| 법령 recall@3 | 76.0% | **92.0%** |
| 법령 recall@1 | 64.0% | **68.0%** |
| 조문 | 별도 경로(미출하) | **응답이 처음부터 조문 단위** (acc@3 58.3%) |

예: `부당해고 구제신청 기간` → 근로기준법 **제28조** / `공소시효 기간` → 형사소송법 **제249조** /
`상속 한정승인 신고 기간` → 민법 **제1019조**.

**비교가 우리에게 불리한 쪽으로 기울어 있다는 점이 중요하다** — 우리 76% 는 이 dev 셋으로
파라미터를 고른 값이고, `aiSearch` 92% 는 이 셋을 본 적 없는 값이다.

이는 LB1~LB5 의 방향 자체를 재고하게 만든다. 우리는 "관련도 랭킹이 없는 upstream" 을 전제로
후보 생성기를 직접 만들어 왔는데, **관련도 랭킹이 있는 엔드포인트가 따로 있었다.**
(단, `aiSearch` 도 만능은 아니다 — "정당방위 성립 요건"에 정당법·방위사업법이 1·2위로 나온다.
즉 대체가 아니라 **1순위 후보 생성기로 교체하고 기존 경로를 보완으로 돌리는** 구도가 유력하다.)

종료가 정당하게 이루어진 것으로 해고가 존재하지 않는다고 판정한 사례".

> (LB1 20% → LB5 80%), 부당해고·구제신청 실무는 법원 판례가 아니라 **노동위원회 판정**에서
> 다뤄진다. 3.9만 건이 통째로 사각지대였다.

## B. 응답 없음 — 존재하지 않거나 파라미터 미상

`lsHistory`(HTML 반환) · `couseLs` · `drlaw` · `lsRlt` · `datPop` · `lawAppr` · `lnkLs`(검색은 2건,
조회는 빈 응답) · `lsHstInf`/`lsJoHstInf`(컨테이너는 있으나 0건 — ID 파라미터 필요 추정) ·

**주의**: 빈 문자열 응답 = "그 이름의 서비스가 없다"가 아니라 **"이 호출 형태로는 안 된다"** 이다.
파라미터가 다르거나(ID/MST/trmSeqs), 조회 전용이거나, 별도 신청이 필요할 수 있다.

## D. 이 조사가 뒤집는 기존 판단

1. **LB3 의 "법원 5종" 은 전수가 아니었다.** 위원회 결정문 6종·별표 3종·법령 변형 4종이 더 있었다.
   당시 "누락 법원 5종 도구화" 라는 milestone 이름 자체가 갭을 과소평가한 것이다.
2. **LB5 의 `dlytrm` 제외 판정은 근거가 부분적이었다.** 검색은 정상이고 관계 조회만 불안정했다.
3. **노동 도메인 저점수의 원인 해석이 불완전했다.** 용어 갭만 봤는데, 실은 **자료원 자체가
   없었다**(노동위 판정 3.9만 건 미접근). 용어 부스트로 20%→80% 가 됐지만 그건 법령 축이고,
   "실제로 어떻게 판정되나"는 여전히 못 답한다.

## D-2. 공식 문서에만 있고 이 OC 로는 빈 응답인 것 (파라미터·권한 미상)

- **`specialDeccTt` 조세심판원** · `specialDeccKmst` 해양안전심판원 · `specialDeccAcr` ·
  `specialDeccAdap` 소청심사 — 문서에 있으나 `lawSearch.do?target=...&query=` 형태로는 빈 응답.
  **조세심판원은 이 레포의 최초 소비처(세무)와 직결**되므로 파라미터 재확인 가치가 높다.
- **`cgmExpc{부처코드}` 중앙부처 1차 해석 39종** — 부처코드 2종(고용노동부·국세청 추정)으로
  시도했으나 빈 응답. 정확한 코드 체계 확인 필요.
- `custLs*` 맞춤형(분류코드 기반) 6종, 모바일 경량판 13종.

## E. 다음 행동 제안 (사용자 결정 사항)

- **M0 (최우선 — 이번 조사가 바꾼 것)**: `aiSearch` 편입. dev 92% vs 우리 76% 는 무시할 수 없는
  격차이고, 조문까지 함께 온다. 다만 **대체가 아니라 A/B 로** 확인해야 한다(정당방위 사례처럼
  aiSearch 가 틀리는 유형이 있다). 우리가 만든 용어 부스트와 어느 쪽이 상위인지도 재측정 대상.
- **M1**: 위원회 결정문 도구화 — `nlrc` 우선. LB3 의 `source-adapter` 구조를 그대로
  재사용하므로 어댑터 확장 수준. 기여도 게이트는 **도메인별 대표 쿼리**로(LB3 에서 골든셋 측정이
  용도 불일치였음을 이미 배웠다).
- **M2**: `lsDelegated` 위임조문 점프 — `get_law_article` 응답에 "이 조문이 위임한 시행령 조문"을
  덧붙이는 형태. 신규 도구 없이 기존 표면 강화.
- **선행 필요**: 측정 하네스 보수 — 홀드아웃이 은퇴했고(LB5), 측정 노이즈(±1건)가 정량화되지
  않았다. 새 세트 라벨링 없이는 다음 개선의 과적합을 판정할 수 없다.
