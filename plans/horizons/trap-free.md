## 목표

Objective 의 **함정 없음** 축을 양쪽에서 민다 — **사용자가 만나는 함정**(설치 첫 화면에서
`LAW_API_OC` 를 만나고 아무 안내가 없다, 도구 11개 중 4개만 문서화돼 있다, 지시대로 `as_of` 를
쓰면 경로가 끊긴다)과 **우리가 만나는 함정**(평가 문제가 4파일로 흩어져 124건 중 34건만 쓰인다).

한 문장: **처음 붙이는 사람이 막히지 않고, 우리가 가진 문제를 전부 쓸 수 있다.**

## 왜 지금 (직전 horizon 이 드러낸 갭)

`agentic-reach` 는 도달을 올렸지만 자를 신뢰할 수 없는 상태로 닫혔다. 원인을 파고 보니 데이터
문제가 아니라 **보관 형식 문제**였다:

- 골드 세트를 4번 만들었다 — `golden`(40) → `golden-v2`(40, 재라벨) → `golden-tax`(50, 분야 확장)
  → `golden-tax-agentic`(34, **입력 형식 변경**). 앞의 셋은 `query`(라벨 문자열)를 갖고 마지막
  하나만 `context`(자연어 맥락)를 갖는다.
- 그래서 에이전트 하네스는 **124건 중 34건만** 쓸 수 있고 90건이 사장돼 있다.

동시에 설치 관문은 두 horizon 동안 한 번도 안 봤다. README 는 도구 11개 중 4개만 적고 OC 를
어디서 받는지 한 줄도 없다. 상류(법제처)도 절차를 공개하지 않는다
(실측 → `research/2026-07-23-trap-free-install-gate.md`).

세 번째로, 세법 특화의 정중앙에 결함이 있다 — 서버가 에이전트에게 "세금·연도 질문은 `as_of` 를
쓰라"고 지시해 놓고(`src/index.ts:42`), `search_law` 가 준 `law_id` 를 `as_of` 에 넘기면 끊긴다.

세 가지 다 성격이 같다 — **아는 사람만 통과하는 관문**. 그래서 한 horizon 에 담는다.

## 범위 조정 (2026-07-23 사용자 판정)

초안에는 **봉인 회전 기계장치**(홀드아웃을 데이터가 아니라 봉인으로 돌리는 구현)가 있었으나
**삭제**했다. 근거:

- 봉인(문제를 떼어 두고 안 보는 것)은 **점수를 보며 도구를 튜닝할 때만** 값을 한다.
  이번 horizon 에 튜닝 대상이 없다.
- 기존 124건 중 dev 79 는 튜닝에 쓰였고 holdout 45 는 과거 horizon 에서 **이미 개봉**됐다 —
  미개봉 문제가 0개라 장치를 만들어도 담을 게 없다.
- 대신 **규약 ADR** 한 줄로 남긴다: 다음에 문제를 늘릴 때 일부를 처음부터 안 보고 떼어 둔다.
  장치는 그때 필요하면 만든다.

빠진 자리에는 세법 사용자가 체감하는 결함 수리(TF4)를 넣었다.

## 닫는 기준 (선언 — close 에서 `선언 / 실측 / 판정` 대조)

1. **단일 코퍼스**: `bench/corpus.json` 1개에 distinct 124건. `ls bench/golden*.json` 이 빈 결과
   (4파일은 `archive/bench/` 로 이동). 스키마 검증 스크립트 exit 0.
2. **수치 재현**: 통합 후 러너 2종이 통합 전 값을 재현 — 범용 dev `recall@3` >= 88.0%,
   agentic dev `pass^3` >= 90%. 재현 실패는 통합 실패로 판정한다(새 기준선으로 받아들이지 않는다).
3. **형식 중립**: 코퍼스 전 케이스가 `query` 와 `context` 를 둘 다 보유(커버리지 100%).
   유출 탐지기 전건 통과(적발 0), 일부러 유출시킨 문단은 여전히 거부.
4. **주제 무개입 증명**: 신규 작성분의 `expected_article` 이 전부 통합 전 4파일에 이미 존재함을
   스크립트가 검증 — **신규 주제 0건**. 봉인 규약 ADR 이 존재하고 기존 124건이 전부 개봉됐음을 명시.
5. **설치 관문**: OC 미설정 환경에서 MCP 서버를 실제 클라이언트로 기동해 **사람이 읽는 안내**가
   관측된다(로그 첨부). README 가 노출 도구 11개를 전부 문서화하고 법제처 신청 페이지로 링크한다.
6. **`as_of` 체인**: 실 MCP 클라이언트에서 `search_law` → `law_id` → `get_law_article(as_of=...)`
   이 한 체인으로 성공하고, 과거 연도와 현행의 조문 내용이 실제로 다름을 확인한다.
7. `npm test` 전건 통과 · 배포 사본(`custom-mcps`) build + dist 스모크 · **재시작 부채 명시**.

## 미리 쓰는 실패 회고(프리모템) — 이 horizon 은 실패했다. 왜?

1. **통합했더니 수치가 안 맞았고, 그걸 새 기준선으로 받아들였다.** 4파일의 라벨 규약이 미묘하게
   다르다(`golden-tax` 는 `type`·`tax_year`, agentic 은 `expect_abstain` 을 갖는다). 재현 실패를
   "코퍼스가 커져서 그렇다"로 넘기면 과거 비교선이 영구히 끊긴다.
   → **예방**: 닫는 기준 2를 통합 실패 판정으로 박는다. TF1 DoD 에 재현 수치를 못 박는다.
2. **90건 맥락을 쓰다 지쳐 품질이 무너졌다.** 맥락이 정답을 그대로 노출(유출)하거나, 반대로 너무
   모호해 풀 수 없는 문제가 됐다.
   → **예방**: 20건 배치마다 유출 탐지기 전건 + 배치 기준선 `pass^3` < 70% 면 그 배치 재작성.
3. **본법 vs 시행령 라벨 쟁점이 통합 도중에 터졌다.** 규약 없이 124건을 합치면 "대통령령으로
   정한다" 위임 지점의 정답이 파일마다 다르게 굳는다(d05·d09 오답이 전부 시행령이었다).
   → **예방**: TF1 의 첫 step 이 라벨 규약 ADR 이다. 규약 없이 마이그레이션에 손대지 않는다.
   확정 규약: 위임 *근거*를 물으면 본법, 구체 *요건·한도·비율*을 물으면 시행령(2026-07-23 사용자 확정).
4. **설치 안내를 문서만 고치고 끝냈다.** 우리는 이미 OC 가 있어서 함정을 못 본다.
   → **예방**: 닫는 기준 5를 실 MCP 클라이언트 무자격 기동 관측으로 박는다. 문서 diff 는 증거가 아니다.
5. **`as_of` 를 고쳤다고 했는데 실제로는 현행 조문이 왔다.** 시점 해석이 조용히 실패하고 현행으로
   대체되면 테스트는 통과하는데 답은 틀린다.
   → **예방**: TF4 의 실패 검증이 "과거 연도와 현행의 조문 내용이 실제로 다른지"를 본다. 같으면 실패.
