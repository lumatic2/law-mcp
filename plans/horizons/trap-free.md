# HORIZON — 함정 없음 (`trap-free`)

> 생성: 2026-07-23 · ROADMAP marker: `harness:goal id="trap-free"` · 상태: active
> cascade 중간(Horizon). 위계: `OBJECTIVE.md` → **이 문서** → `plans/<date>-<milestone-id>-<slug>.md` → Step.
> 진행 상태 정본 = 각 plan doc 의 step 체크박스. 이 문서의 milestone 체크박스는 boundary 동기화.

## 목표

Objective 의 **함정 없음** 축을 양쪽에서 민다 — **사용자가 만나는 함정**(설치 첫 화면에서
`LAW_API_OC` 를 만나고 아무 안내가 없다)과 **우리가 만나는 함정**(개선을 재려 할 때마다
자를 새로 만들어 과거 비교선이 끊긴다)을 같이 없앤다.

한 문장: **처음 붙이는 사람이 막히지 않고, 우리가 잰 값이 다음 horizon 에도 유효하다.**

## 왜 지금 (직전 horizon 이 드러낸 갭)

`agentic-reach` 는 도달을 올렸지만 **자를 신뢰할 수 없는 상태로 닫혔다.** 홀드아웃이
`pass^5` 100% 로 변별력을 잃었고(원인: 10건 중 6건을 에이전트가 *주제까지* 골랐다),
수치는 사용자 판단으로 미채택됐다. 원인을 파고 보니 데이터 문제가 아니라 **구조 문제**였다:

- 골드 세트를 4번 만들었다 — `golden`(40) → `golden-v2`(40, 재라벨) → `golden-tax`(50, 분야 확장)
  → `golden-tax-agentic`(34, **입력 형식 변경**). 앞의 셋은 `query`(라벨 문자열)를 갖고
  마지막 하나만 `context`(자연어 맥락)를 갖는다.
- 그래서 agentic 하네스는 **124건 중 34건만** 쓸 수 있다. 나머지 90건은 형식이 안 맞아 사장됐다.
- 홀드아웃이 소진될 때마다 **새 데이터를 급조**해 보충해왔고, 그 급조가 이번 오염의 직접 원인이다.

동시에 설치 관문은 두 horizon 동안 한 번도 안 봤다: README 는 도구 11개 중 4개만 적고 있고,
OC 를 **어디서 어떻게 받는지 한 줄도 없다**. 상류(법제처)도 절차를 공개하지 않는다
(실측 → `research/2026-07-23-trap-free-install-gate.md`).

두 문제는 성격이 같다 — **아는 사람만 통과하는 관문**. 그래서 한 horizon 에 담는다.

## 닫는 기준 (선언 — close 에서 `선언 / 실측 / 판정` 대조)

1. **단일 코퍼스**: `bench/corpus.json` 1개에 distinct 124건 이상. `ls bench/golden*.json` 이
   빈 결과(4파일은 `archive/` 로 이동). 스키마 검증 스크립트 exit 0.
2. **수치 재현**: 통합 후 러너 2종이 통합 전 값을 재현 — 범용 dev `recall@3` ≥ 88.0%,
   agentic dev `pass^3` ≥ 90%. 재현 실패는 통합 실패로 판정한다(새 기준선으로 받아들이지 않는다).
3. **형식 중립**: 코퍼스 전 케이스가 `query` 와 `context` 를 **둘 다** 보유(커버리지 100%).
   유출 탐지기 전건 통과(적발 0), 일부러 유출시킨 문단은 여전히 거부.
4. **봉인 회전 동작**: 미개봉 슬라이스는 플래그 없이 `exit 1`. 슬라이스를 회전시킨 뒤에도 동일.
   개봉 이력이 파일로 남아 같은 슬라이스를 두 번 개봉하면 거부.
5. **주제 무개입 증명**: 신규 작성분의 `expected_article` 이 전부 통합 전 4파일에 이미 존재함을
   스크립트가 검증 — **신규 주제 0건**. (이번 horizon 이 오염을 반복하지 않았다는 기계 증거.)
6. **설치 관문**: OC 미설정 환경에서 MCP 서버를 실제 클라이언트로 기동해 **사람이 읽는 안내**가
   관측된다(로그 첨부). README 가 노출 도구 11개를 전부 문서화하고 발급 절차를 담는다.
7. `npm test` 전건 통과 · 배포 사본(`custom-mcps`) 반영 절차 명시.

## 미리 쓰는 실패 회고(프리모템) — 이 horizon 은 실패했다. 왜?

1. **통합했더니 수치가 안 맞았고, 그걸 새 기준선으로 받아들였다.** 4파일의 라벨 규약이 미묘하게
   다르다(`golden-tax` 는 `type`·`tax_year`, agentic 은 `expect_abstain` 을 갖는다). 재현 실패를
   "코퍼스가 커져서 그렇다"로 넘기면 과거 비교선이 영구히 끊긴다.
   → **예방**: 닫는 기준 2를 통합 실패 판정으로 박는다. TF1 DoD 에 재현 수치를 못 박는다.
2. **90건 맥락을 쓰다 지쳐 품질이 무너졌다.** 맥락이 정답을 그대로 노출(유출)하거나, 반대로 너무
   모호해 풀 수 없는 문제가 됐다.
   → **예방**: 배치마다 유출 탐지기 전건 + 배치 기준선 `pass^3` 이 기존 dev(90%) 대비 급락(<70%)
   하면 그 배치를 재작성한다. 배치 단위로 검증해 끝에 몰아서 발견하지 않는다.
3. **본법 vs 시행령 라벨 쟁점이 통합 도중에 터졌다.** 규약 없이 124건을 합치면 "대통령령으로
   정한다" 위임 지점의 정답이 파일마다 다르게 굳는다(d05·d09 오답이 전부 시행령이었다).
   → **예방**: TF1 의 **첫 step 이 라벨 규약 ADR**이다. 규약 없이 마이그레이션에 손대지 않는다.
4. **설치 안내를 문서만 고치고 끝냈다.** 우리는 이미 OC 가 있어서 함정을 못 본다.
   → **예방**: 닫는 기준 6을 실 MCP 클라이언트 무자격 기동 관측으로 박는다. 문서 diff 는 증거가
   아니다.
