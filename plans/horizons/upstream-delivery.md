# HORIZON — 법제처가 가진 능력을 그대로 전달 (upstream-delivery)

> 생성: 2026-07-21 · ROADMAP marker: `harness:goal id="upstream-delivery"` · 상태: active
> cascade 중간(Horizon). 위계: Objective(`OBJECTIVE.md`) → **Horizon**(이 문서) → Milestone(`plans/<date>-<milestone-id>-<slug>.md`) → Step.
> 진행 상태의 정본은 `ROADMAP.md` marker — 이 문서의 체크박스는 milestone boundary 에서만 동기화.

## 목표

**우리가 랭킹을 만드는 대신, 법제처가 이미 가진 것을 손실 없이 전달한다.**

직전 horizon 은 "upstream 에 관련도 랭킹이 없다"를 전제로 후보 생성기를 5 milestone 에 걸쳐
직접 만들었다. 그 전제가 틀렸다 — 랭킹이 있는 엔드포인트(`aiSearch`)가 따로 있었고, 무튜닝으로
우리 튜닝값을 이긴다. 이 horizon 은 축을 바꾼다:

| | 구 축 (general-legal-coverage) | **신 축 (이 horizon)** |
|---|---|---|
| 품질의 정의 | 우리 랭킹이 정답을 상위에 올리는가 | **소비 LLM 이 필요한 자료원에 도달하는가 + 응답이 그대로 인용 가능한가** |
| 개선 수단 | 후보 생성·재정렬 알고리즘 | **더 나은 upstream 엔드포인트 선택 + 충실한 래핑** |
| 실패 모드 | 랭킹이 나쁘다 | **가진 걸 안 쓴다 / 있는 걸 깎아서 전달한다** |

이 축에서는 `nlrc` 3.9만 건이 아예 없는 것, `lsDelegated` 가 이미 계산해 둔 위임 점프를 안 쓰는
것이 랭킹 4%p 보다 훨씬 큰 결함이다.

## 왜 지금

전수 조사 `research/2026-07-21-lawgo-api-survey.md`(실 API 프로브 83종 + 공식 가이드 191건):

- **`aiSearch` 가 우리보다 낫다** — 같은 dev 셋에서 recall@3 **92.0% vs 76.0%**, recall@1 68% vs 64%,
  게다가 응답이 **처음부터 조문 단위**다(`부당해고 구제신청 기간` → 근로기준법 제28조).
  비교는 우리에게 유리하게 기울어 있다(우리 76% 는 이 셋으로 튜닝한 값, 92% 는 무튜닝).
- **우리가 쓰는 것 8종 / 문서가 나열하는 것 191건.** 실 API 로 실재 확인한 미사용만 25종 이상.
- **노동 도메인 저점수의 진짜 원인은 자료원 부재였다** — 부당해고 실무는 법원 판례가 아니라
  노동위원회 판정(`nlrc` 39,363건)에서 다뤄지는데, 그게 통째로 사각지대였다.
- **측정 도구가 지금 고장나 있다** — 홀드아웃 15건은 LB5 에서 소진·은퇴했고, 노이즈는 ±1건(4%p)
  인데 정량화되지 않았다. 이대로 `aiSearch` 를 편입하면 "정말 좋아졌나"를 판정할 수 없다.

이 horizon 이 이 세 가지를 순서대로 친다: **측정 복구 → 더 나은 엔진 편입 → 없던 자료원 흡수.**

## 무감독 분량

**승인 후 최소 3 무감독 세션.** 근거: milestone 3개 / 예상 changeset 합계 ≥10.
UD1 은 세트 라벨링·반복측정 러너·기준선 재측정, UD2 는 클라이언트·후보 편입·조문 출하·판정,
UD3 은 위원회 어댑터·위임조문·기여도 게이트. UD1→UD2 는 강한 의존(측정 없이는 편입 판정 불가),
UD2→UD3 은 독립이라 연쇄가 끊기지 않는다.

## 미리 쓰는 실패 회고 (프리모템)

이 horizon 이 끝났는데 실패했다. 왜?

1. **`aiSearch` 로 갈아탔더니 특정 유형에서 크게 나빠졌다.** 92% 는 평균이고, 조사에서 이미
   "정당방위 성립 요건 → 정당법·방위사업법" 이라는 반례를 봤다. → 예방: UD2 는 **대체가 아니라
   병합**으로만 편입하고, **쿼리 단위 승패 표**(어떤 쿼리가 좋아지고 나빠졌나)를 DoD 에 넣는다.
   집계 수치만 보고 채택하지 않는다. 새로 깨지는 쿼리가 0 이 아니면 병합 규칙을 고친다.
2. **upstream 블랙박스에 품질을 인질로 잡힌다.** `aiSearch` 는 법제처가 언제든 바꿀 수 있고
   우리는 이유를 모른다. → 예방: 기존 경로를 **삭제하지 않고 폴백으로 보존**하고, `aiSearch`
   무응답·타임아웃 시 graceful degrade 를 failure probe 로 고정한다. 두 경로 모두 테스트에 남긴다.
3. **새 평가 세트가 또 오염된다.** UD1 에서 만든 세트를 UD2 튜닝에 쓰고, UD3 에서 또 열어보면
   LB5 와 같은 자리로 돌아온다. → 예방: UD1 에서 **dev / holdout 을 분리 라벨링**하고 holdout
   열람 시점을 plan 에 **미리 못박는다**(horizon close 시 1회). 중간 A/B 는 dev 로만 한다.
4. **자료원을 늘렸는데 도구가 늘어 선택 부담만 커진다.** LB3 에서 이미 겪었다. → 예방: UD3 는
   `source` enum 확장으로만 처리하고 **도구 표면 증가 ≤1** 을 닫는 기준에 박는다(위임조문은
   기존 `get_law_article` 응답 강화, 신규 도구 없음).
5. **또 "있는 걸 뒤늦게 발견"한다.** 두 번 겪은 실수다. → 예방: 이 horizon 은 전수 조사를
   **선행**해서 연다(부채 상환 완료). 각 milestone 은 그 조사 문서의 해당 절을 소비처로 백링크한다.
