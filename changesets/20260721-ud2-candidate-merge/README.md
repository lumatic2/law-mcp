# UD2 step-2 — `aiSearch` 후보 병합 + 교차 A/B 판정

- 일자: 2026-07-21 · milestone: UD2 · plan: `plans/2026-07-21-ud2-aisearch-adoption.md`
- 증거: `evidence/bench/2026-07-21-ud2-ab.md`

## 무엇이 달라지나

주제 검색(`search_law`)이 **법제처가 매긴 관련도 순위를 1순위 신호로** 쓴다. dev 25건에서
정답 포함률 48% → 80%, **새로 깨진 쿼리 0건**.

기존 사다리(이름 → 본문 → 완화 → 용어 브리지)와 용어 연계 부스트는 **그대로 남는다.**
`aiSearch` 를 앞단 채널로 얹었을 뿐이라, 편입 지점 1곳을 끄면 LB5 상태로 즉시 복귀한다.

## 왜 대체가 아니라 병합인가 — 값으로 확인했다

`aiSearch` 단독도 recall@3 80% 로 병합과 **집계가 같다.** 쿼리 단위로 봐야 갈렸다:
`사용자책임 면책 사유` 를 기존 사다리는 맞히는데 `aiSearch` 는 놓친다. 단독 채택이었으면
**이게 회귀로 출하됐다.** 집계 지표만 봤으면 구분하지 못했을 판정이다.

우선순위(`aiSearch` 우선 vs 부스트 우선)도 선험적으로 정하지 않고 실측으로 골랐다 — +8 대 +7.

## 판정 규약 (F13)

배치 A 전체 → 배치 B 전체가 아니라 **쿼리마다 4개 배치를 연달아** 쟀다. 이 벤치의 문제는
노이즈가 아니라 드리프트라서(UD1), 순차 측정은 드리프트를 나중에 잰 쪽의 불이익으로 만든다.
채택 조건 = **새로 깨지는 쿼리 0 AND 순 이득 ≥3건**. 2회 측정이 한 건도 다르지 않았다.

## 회귀 방어 (UD0 교훈)

순위를 바꾸는 변경은 **그 순위를 입력으로 쓰는 도구까지** 회귀 범위다. LB5 가 이걸 안 해서
`get_law_article` 이 다른 법의 조문을 반환했다. 그래서 `resolveLawId`(법령명 → ID)는
`aiSearch` 를 명시적으로 끄고, 그 사실을 테스트로 잠갔다.

## 적발한 것 — 테스트가 실 API 를 타고 있었다

기본값을 켜는 순간 `term-boost.test.ts` 4건이 실패했다. 원인은 그 테스트들이 `aiSearch` 를
주입하지 않아 **진짜 법제처를 호출**하고 있었다는 것. 파일 상단에 "실 API 를 타지 않도록
주입한다"고 적혀 있었는데 새 채널이 그 격리를 뚫었다. 빈 신호로 고정해 격리를 복구했다.
