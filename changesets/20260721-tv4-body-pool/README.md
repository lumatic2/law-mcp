## 무엇을 확인했나 — 절단은 실재했다

본문검색 응답은 **관련도순이 아니라 가나다순**이다. 앞 30건만 받으면 뒷글자 법령이 구조적으로
탈락한다. 세법 dev 30건 실측(2026-07-22):

| 후보 풀 창 | 정답이 풀에 드는 비율 |
|---|---|
| 30건 (TV3 상태) | **19/30 (63%)** |
| 100건 | **29/30 (97%)** |

절단이 실제로 일어나는 쿼리가 **16/30**이다. `수정신고 할 수 있는 경우` 는 469건,
`심판청구 청구기간` 은 422건인데 우리는 30건만 보고 있었다.

## 그런데 넓혀도 recall 은 안 움직였다 → **미채택**

| 배치 | recall@3 | 이득 | 손실 | 순 | 지연(중앙) |
|---|---|---|---|---|---|
| TV3 상태 (풀1p·신호 off) | 80.0% | — | — | — | 1879ms |
| 풀 도달만 (풀3p) | 80.0% | 0 | 0 | **+0** | 1917ms |
| 조문제목 신호만 | 80.0% | 1 | 1 | **+0** | 2728ms |
| 풀 도달 + 조문제목 신호 | 80.0% | 1 | 1 | **+0** | 2637ms |

**정답이 후보에 들어와도 위로 못 올라온다.** 순위를 정하는 신호가 법령명 문자열뿐이라,
300건짜리 풀에서 `국세기본법` 은 이름이 짧은 다른 법들에 밀려 여전히 3위 밖이다. 도달과
순위는 **다른 문제**였고, 이 milestone 은 도달만 고쳤다.

채택 규약(**손실 0 AND 순 이득 ≥2**)을 못 넘겼으므로 **기본값은 TV3 그대로 둔다**
(`BODY_POOL_DEFAULT_PAGES = 1`). 넓히기는 호출부가 켤 때만 동작한다.

## 그래도 남기는 것 — 순위가 아니라 **정직성**의 문제

미채택은 재정렬·풀 확대에 대한 판정이다. 아래 둘은 순위를 바꾸지 않고 **틀린 것을 고친다**:

### ① `totalCnt` 를 잘못 읽고 있었다 (잠복 버그)

`totalCnt` 는 `LawSearch` 컨테이너 **안에** 있는데 root 에서 찾고 있었다. 항상 null 이 나와
`total` 이 조용히 **"받아 온 행 수"** 로 대체됐다 — 실제 469건인 쿼리가 `total: 100` 으로
나가고 있었다. 소비 LLM 이 "총 100건 중 다 봤다"고 읽을 수 있는 값이다.

### ② 절단 경고 — 조용한 절단이 이 결함의 원인이었다

전체 건수가 확인한 건수보다 많으면 경고를 싣는다. 실 API 관측:

> "본문검색 전체 **469건** 중 앞 **300건**만 확인함(upstream 순서는 관련도가 아니라 가나다순)
> — 찾는 법령이 안 보이면 쿼리를 좁혀 다시 물을 것."

사용자가 목록을 다 봤다고 믿고 "그런 법은 없다"고 결론하는 것이 이 결함의 실제 피해였다.
경고는 순위를 바꾸지 않으므로 채택 규약의 대상이 아니다.
