## LB3 과 다른 점 — 이번엔 규칙이 있다

LB3 법원 5종은 키가 제각각이라(`ordin` 의 행 키가 `law`) 하나씩 못 박아야 했다. 위원회 계열은
**9/9 전부 규칙이 일정**하다: 컨테이너 = 타깃명 대문자, 행 키 = 타깃명, 단건 = `<컨테이너>Service`,
조회 파라미터 = `ID`. 그래서 팩토리 하나로 쓰고, **규칙 자체를 테스트로 잠갔다** — 새 위원회를
실측 없이 추측으로 넣는 것도 그 테스트가 막는다.

조회 파라미터 실패 모드도 확인했다: `nlrc` 에 `MST` 나 없는 번호를 주면
`{"Law":"일치하는 결정문이 없습니다..."}` 가 온다. **`ordin` 처럼 조용히 다른 문서를 주지 않는다** —
LB3 보다 안전한 계열이다. 그래도 파라미터는 못 박았다.

## ★ 스모크가 드러낸 기존 결함 — `limit: 1` 이면 모든 법원이 0건이었다

9종을 붙이고 스모크를 돌리자 **전부 0건**이 나왔다. 직접 프로브는 되는데 우리 경로만 안 됐다.

원인: **결과가 1건이면 배열이 아니라 객체로 온다.** `display=1` → 단일 객체, `display=2` → 배열.
`extractRows` 가 `Array.isArray` 만 봐서 단건을 못 읽었다.

**이건 위원회만의 문제가 아니라 LB3 이후 출하돼 있던 결함이다** — `expc` 도 `limit:1` 이면 0건이었다.
검색이 실패한 게 아니라 우리가 못 읽은 것이라 **경고조차 뜨지 않았다.** 이번에 함께 고쳤다.
