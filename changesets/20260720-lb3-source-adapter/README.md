## 무엇을 했나

신규 법원 5종을 얹기 전에, ① **타깃↔법원 대응을 실 API 응답 필드로 확정**하고 ② 검색 응답 파싱과
폴백 사다리를 공통 헬퍼로 뽑았다. 도구는 아직 등록하지 않는다(step-2·3).

## ① 타깃 확정 — 이름이 아니라 응답 필드로 (2026-07-21 실 API 프로브)

| target | 실제 법원 | 컨테이너 | 행 배열 키 | 확정 근거(응답 필드) |
|---|---|---|---|---|
| `expc` | 법령해석례 | `Expc` | `expc` | `법령해석례일련번호`·`안건명`·`회신기관명` |
| `detc` | 헌재결정례 | `DetcSearch` | `Detc` | `헌재결정례일련번호`·`사건번호`(2014헌마215) |
| `decc` | 행정심판재결례 | `Decc` | `decc` | `행정심판재결례일련번호`·`재결청`·`재결구분명` |
| `ordin` | 자치법규 | `OrdinSearch` | **`law`** | `자치법규ID`·`자치법규명`·`지자체기관명` |
| `lstrm` | 법령용어 | `LsTrmSearch` | `lstrm` | `법령용어ID`·`법령용어명` |

**컨테이너·행 키에 규칙이 없다**는 것이 이번 프로브의 핵심 관측이다. 대소문자가 제각각이고
(`Expc` vs `DetcSearch` vs `Decc`), 특히 `ordin` 은 행 키가 타깃명이 아니라 `law` 다 —
이름으로 추측해 파싱했다면 자치법규는 **에러 없이 조용히 0건**이 됐을 것이다.

plan 이 경고한 "detc/decc 가 직관과 반대" 도 확인됐다. 이름만 보면 `decc`("decision")가 헌재
같지만, 응답 필드는 `detc`=헌재, `decc`=행정심판이라고 말한다. 테스트가 이 대응을 응답 필드로
고정한다(`detc/decc are pinned to the source their response fields prove`).

## ② 공통 어댑터 `src/providers/source-adapter.ts`

- `SOURCE_DESCRIPTORS` — 5종의 실측 키·ID 필드·제목 필드·본문검색 지원 여부 선언.
- `extractRows` — 선언 키 → 컨테이너 내 "객체 배열" → 최상위 컨테이너 순 폴백. **못 찾으면 예외가
  아니라 0건 + 경고**다. 설계 의도: upstream 스키마 변경이 도구 크래시가 돼도 안 되고, 조용한
  0건이 돼서도 안 된다. 정상 0건(`totalCnt=0`)과 구조 미상은 경고 유무로 구분한다.
- `mapRow` — 법원별 필드를 `source_id`/`title` + 부가 필드로 표준화.
- `searchWithLadder` — 이름 → 본문(search=2) → 완화 → 용어 브리지 → 브리지+완화 사다리를
  파라미터화. `supportsBodySearch: false`(법령용어)면 mode 2 를 아예 쓰지 않고 "관련도 정렬 아님"
  경고도 붙이지 않는다.

## 테스트가 잡은 내 오류

사다리 순서를 "이름 → 본문 → 완화 1단 → **완화 2단**"으로 기대하는 테스트를 썼는데 실패했다.
실제 계약은 완화를 **1단만** 하고 바로 용어 브리지로 넘어가는 것이다(다단 완화는 브리지 이후
`bridgeThenRelaxSearch` 담당). 원본 구현이 옳고 내 기대가 틀렸다 — 기대치를 실제 계약에 맞췄다.

## 관측 (step-3 기여도 게이트로 넘김)

**법령용어(`lstrm`)는 개념어 사전이 아니다.** "정당방위" 는 0건이고, "소득"·"근로자" 는
"2011년 12월 31일에 퇴직하였다고 가정할 때 지급받을 대통령령으로 정하는 퇴직소득금액" 같은
**법령이 정의한 긴 문구**를 돌려준다. plan 의 finding F1(`term-bridge.ts` 수기 사전을 lstrm 으로
대체) 은 이 관측만으로도 회의적이고, 기여도 게이트에서 미등록될 가능성이 높다. 판단은 step-3 에서
수치로 한다.
