- 일자: 2026-07-21 · horizon: `upstream-delivery` · 성격: **핫픽스**(LB5 회귀)
- 발견 경위: UD1 step-1 평가 세트 라벨을 실 API 로 검증하다 적발

## 무엇이 잘못됐나

`get_law_article("민법", "제245조")` 가 **새마을금고법 조문**을 HTTP 200 으로 반환했다.
오류가 아니라 **틀린 답**이라 아무도 못 봤다.

실측 재현:

```
민법       ON : 새마을금고법 / 수의사법 / 민법[exact]
           OFF: 민법[exact] / 민법법인 등기규칙 / …
행정심판법  ON : 광업법 / 국민연금법 시행령 / 행정심판법[exact]
근로기준법  ON : 보호소년 등의 처우에 관한 법률 / 근로기준법[exact] / …
```

라벨 검증기가 "OK"로 통과시킨 본문들:
`근로기준법 제37조 :: 통근취업`(보호소년법) · `행정심판법 제27조 :: 출원인의 명의 변경`(광업법) ·
`형법 제48조 :: 징계처분`(법무사법).

## 원인

`resolveLawId` 가 `searchLaw(name).items[0]` 을 **정확일치 요구 없이** 그대로 썼다.
LB5 가 붙인 용어 연계 부스트는 쟁점형 질의용인데, 법령명을 그대로 물으면 정확일치를 1위에서
밀어낸다. 그 1위가 `get_law_article`·`verify_citation` 의 **입력**이 되면서 도구가 통째로
엉뚱한 법을 가리켰다.

**LB5 의 "새로 깨지는 쿼리 0" 게이트가 이걸 놓친 이유**: 그 측정은 `search_law` 의 *순위*만 봤고,
순위가 *다른 도구의 입력*이 된다는 걸 보지 않았다. 골든셋 쿼리도 전부 쟁점형이라
"법령명을 그대로 묻는" 유형은 한 번도 측정되지 않았다.

## 수리

이 경로는 **이름 조회**이지 주제 검색이 아니다. 두 가지를 지킨다:

1. 용어 연계 부스트를 **끈다** — 이름 조회에는 순위 신호가 필요 없다.
2. **정확일치 → prefix → 첫 항목** 순으로 고른다(`pickLawIdByName`, 순수 함수로 분리해 테스트 가능).

부스트 자체는 `search_law` 순위에 그대로 남는다 — LB5 의 이득(recall@3 44%→76%)은 보존된다.
