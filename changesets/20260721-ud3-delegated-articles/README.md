# UD3 step-2 — 위임조문 점프

- 일자: 2026-07-21 · milestone: UD3 · plan: `plans/2026-07-21-ud3-untapped-sources.md`

## 무엇이 안 되던 것

법률 조문의 **"대통령령으로 정하는 바에 따라"** 를 만나면 거기서 끊겼다. 소비 LLM 은 그 다음이
어느 시행령 몇 조인지 스스로 찾아야 했고, 대개 못 찾았다.

법제처는 그 계산을 이미 해 두고 있었다(`lsDelegated`). 소득세법 하나에 **189개 조문 / 365건**의
위임 관계가 들어 있다.

## 붙인 것

`get_law_article` 응답에 `delegated_to` 선택 필드. **신규 도구 없음**, 도구 개수 11개 그대로.

```
소득세법 제1조의2(정의)
  → 소득세법 시행령 제2조(주소와 거소의 판정)
  → 소득세법 시행령 제2조의2(거주자 또는 비거주자가 되는 시기)

소득세법 제70조(종합소득과세표준 확정신고)
  → 소득세법 시행령 제130조(종합소득 과세표준확정신고)
  → 소득세법 시행규칙 제102조(조정계산서 관련서식)
  → 소득세법 시행령 제131조의2(외부세무조정 대상사업자의 범위)
  → 소득세법 시행령 제131조의3(조정반)
```

위임이 없는 조문(민법 제245조 등)에는 **필드를 아예 달지 않는다** — 빈 배열 오염 금지.

## ★ 이 step 이 스스로 잡은 결함 — 23건이 조용히 사라지고 있었다

첫 구현으로 스모크를 돌리니 `소득세법 제70조` 에 위임이 없다고 나왔다. 원문에는 `70` 이 있었다.
"없는 게 맞나"를 확인하지 않았으면 그대로 넘어갔을 값이다.

원인: **`위임정보` 자체가 배열일 수 있다** — 한 조문이 시행령과 시행규칙에 **동시에** 위임할 때
그렇다. 객체로만 읽어서 그런 항목이 통째로 빠졌다.

| | 수리 전 | 수리 후 |
|---|---|---|
| 위임 보유 조문 | 166 | **189** (=전량) |
| 위임 건수 | 289 | **365** |

이 레포에서 오늘만 **세 번째** 배열/객체 흔들림 결함이다(UD3 step-1 `extractRows`, 여기 두 곳).
DRF 는 "1건이면 객체"를 곳곳에서 한다 — 이제 새 파서를 쓸 때 기본 의심 대상이다(F16).
