## 무엇이 달라지나

검색이 **답을 반쯤 찾고 멈추던 것**을 끝까지 준다. dev recall@3 80% → **88%**.

```
매입세액 불공제 대상
  전: 부가가치세법 시행령 / 조세특례제한법 / 부가가치세법 시행규칙
  후: 부가가치세법 / 조세특례제한법 / 부가가치세법 시행규칙
```

## 규칙

한국 법령의 하위법령은 이름이 `<본법명> 시행령` 꼴이라 본법명을 문자열로 복원할 수 있다.
복원한 이름이 **실재하는지 조회로 확인한 뒤에만** 편입한다 — 이름 규칙만 믿으면 없는 법을
그럴듯하게 반환하는 UD0 급 오답이 된다.

**법명·도메인·쿼리 토큰이 하나도 안 들어간다.** dev 2건을 보고 만들었지만 규칙은 이름 형태만
본다. 법명을 하드코딩하는 순간 그건 평가 세트에 맞춘 것이지 검색이 좋아진 게 아니다(F5).

## ★ 규칙을 한 번 갈아엎었다 — 끼워 넣기 → 자리 바꾸기

첫 구현은 본법을 **맨 앞에 끼워 넣었다.** A/B 결과 이득 2 / **손실 1** — 채택 불가.
`사용자책임 면책 사유` 에서 3위였던 정답 민법이 목록 끝에서 잘려 나갔다. 끼워 넣기는
limit 에서 잘리는 목록의 **하위 순위를 밀어낸다**.

그래서 **그 하위법령의 자리를 본법이 차지**하도록 바꿨다. 목록 길이가 그대로라 아무도 안
밀린다. 하위법령은 본법에서 이어서 찾아갈 수 있지만, 밀려난 정답은 소비 LLM 에게 아예 없다.

재측정: 이득 2 / **손실 0** → 채택. 2회 측정 결과 동일.

## 채택 문턱

UD2 의 ≥3 을 **≥2 로 낮췄다.** 겨냥한 결함 유형이 dev 에 2건뿐이라 ≥3 은 달성 불가능한
문턱이다. **손실 0 은 낮추지 않았다.** 러너에 `--min-net` 을 빼서 문턱이 코드에 박히지 않게 했다.
