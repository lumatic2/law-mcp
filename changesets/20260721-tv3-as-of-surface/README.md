# TV3 step-2 — 표면 전파 (`as_of` 인자 + `effective_date` 상시 출하)

- 일자: 2026-07-21 · horizon: `tax-vertical` · milestone: TV3
- plan: `plans/2026-07-21-tv3-tax-year-axis.md` step-2

## 무엇이 달라지나

`get_law_article` 이 **그때의 조문**을 준다. 같은 조문 번호가 연도마다 다르다는 것이 실측으로 보인다:

```
get_law_article("소득세법", "제89조", as_of="2019") → effective_date=20190101  본문 1457자
get_law_article("소득세법", "제89조", as_of="2023") → effective_date=20230701  본문 1335자
get_law_article("소득세법", "제89조")                → effective_date=20260101  본문 1335자
```

그리고 **시점을 지정하지 않아도 `effective_date` 가 항상 실린다.** 조문을 인용하는 쪽에
시행일이 없는 것이 검색 결과에 없는 것보다 위험하다.

시점을 지정하면 `as_of_rule` 도 함께 나간다 — 우리가 그 연도를 어떻게 해석했는지를 사용자가
검증할 수 있어야 한다:

```
as_of_rule: "2023년 귀속으로 보고 그 해 12월 31일 시점에 시행 중이던 법령을 조회했다.
             역년 과세(소득세·부가가치세 등) 기준이며, 사업연도가 역년과 다른 법인세 등은
             정확한 시행일을 직접 지정해야 한다."
```

## 못 맞추면 거절한다 — 현행으로 대체하지 않는다

닫는 기준 3("조용한 현행 반환 0건")이 여기서 결정된다.

```
as_of="작년"  → INVALID_ARGUMENT      "…해석할 수 없는 시점에 현행 법령을 대신 주지 않는다."
as_of="1900"  → INVALID_ARGUMENT
as_of="1955"  → LAW_VERSION_NOT_FOUND "…확인된 가장 이른 시행일은 20111013 이다.
                                        현행 법령을 대신 주지 않는다 — 잘못된 연도의 조문은 오답이다."
```

## 잡은 결함 — 가드에 타깃을 하드코딩했다

`if (efYd) assertEffectiveDateTarget("law")` 로 써서 **시점 조회가 항상 던졌다.** 의도는
"왜 law 를 안 쓰는지 코드로 남긴다"였는데, 실제로는 정상 경로를 전부 막았다. 실 API 프로브에서
첫 호출이 그대로 터져 즉시 드러났다 — `assertEffectiveDateTarget(target)` 으로 고쳤고,
같은 실수를 막는 테스트를 넣었다(`assertEffectiveDateTarget("law")` 하드코딩 금지).
