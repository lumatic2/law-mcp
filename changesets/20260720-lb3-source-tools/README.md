# LB3 step-2 — 법원별 search/get 구현

- 일자: 2026-07-21 · milestone: LB3 · plan: `plans/2026-07-20-lb3-missing-sources.md`
- 증거: `evidence/bench/2026-07-21-lb3-source-smoke.txt`

## 무엇을 했나

5종 법원의 검색·단건조회를 **메서드 2개**로 구현했다(법원마다 2벌씩 10개가 아니라).
`searchLegalSource(target, query)` / `getLegalSource(target, id)` 가 디스크립터를 받아 동작한다.
**도구 등록은 하지 않았다** — step-3 기여도 게이트 통과분만 노출한다.

## 단건 조회에서 나온 함정 3종 (전부 조용한 오답 유발)

### ① `ordin` 은 `ID` 가 아니라 `MST` 로 조회해야 한다 — 틀리면 **다른 조례가 200 으로 온다**

가평군 조례(`자치법규일련번호=2124591`)를 `ID=2124591` 로 조회하면 에러가 아니라
**창원시 하수도 사용 조례 시행규칙**(`자치법규ID=2124592`)이 정상 응답으로 돌아온다.
404 도 빈 응답도 아니어서 호출자가 오답을 알아챌 방법이 없다. 디스크립터에 `idParam` 을
못 박고 테스트로 고정했다(`every source pins its detail lookup parameter`).

### ② 단건 컨테이너가 검색 컨테이너와 다르고, 이름이 겹치기까지 한다

| target | 검색 컨테이너 | 단건 컨테이너 |
|---|---|---|
| expc | `Expc` | `ExpcService` |
| detc | `DetcSearch` | `DetcService` |
| decc | `Decc` | **`PrecService`** (판례와 같은 이름) |
| ordin | `OrdinSearch` | `LawService` (법령과 같은 이름) |
| lstrm | `LsTrmSearch` | `LsTrmService` |

같은 문서의 일련번호 필드명도 검색(`행정심판재결례일련번호`)과 단건(`행정심판례일련번호`)이 다르다.

### ③ 자치법규 조문 구조가 법령과 다르다

법령은 `조문.조문단위` + `조문번호`(문자열) + `조문여부: "조문"` 인데, 자치법규는
`조문.조` + `조문번호`(**배열** `["000100","000100"]`) + `조문여부: "Y"` 다. 기존
`extractArticles` 를 그대로 썼다면 0건이 됐다. `articlesPath` 로 분리해 처리한다.

## 앞선 내 판단 정정 — 법령용어(`lstrm`)

step-1 changeset 에서 "lstrm 은 개념어 사전이 아니라 법령 정의 문구만 준다"고 적었는데,
**부분적으로 틀렸다.** "퇴직소득" 검색에서 3위로 `[20750] 퇴직소득` 이라는 실제 개념어 항목이
나왔고 정의문도 온전하다. 다만 정렬이 긴 정의 문구를 위로 올리고 "정당방위" 는 여전히 0건이라,
**형사·민사 개념어 커버리지는 여전히 의심스럽다.** 확정 판단은 step-3 기여도 수치로 한다.
