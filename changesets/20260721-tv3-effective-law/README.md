# TV3 step-1 — 시점 조회 경로 (`src/effective-law.ts`)

## 실패 프로브가 설계 가정을 뒤집었다

계획 단계의 프로브는 `eflaw&efYd=20230101` 로 2023년 소득세법(338조)을 받았고, 나는 거기서
"`efYd` = 이 날짜 시점"이라고 읽었다. **틀렸다.**

```
eflaw&LM=소득세법&efYd=20231231  →  조문 0개
eflaw&LM=소득세법&efYd=20230101  →  조문 338개
```

`efYd` 는 **"시행일자가 정확히 이 날인 판"** 이다. 앞선 프로브가 통한 건 `20230101` 이 우연히
실제 시행일자였기 때문이다. **기준일을 그대로 넘기는 구현이었다면 대부분의 연도에서 조용히
0 조문을 받았을 것이다.**

그래서 시점 조회를 **두 단계**로 고쳤다:
① `eflaw` 검색으로 그 법의 시행판 목록을 받고 ② 기준일 이하의 **가장 최근 판**을 골라 조회한다.

```
as_of=2019 (기준일 20191231) → 시행판 20190101 · 조문 319개
as_of=2023 (기준일 20231231) → 시행판 20230701 · 조문 339개
as_of=2024 (기준일 20241231) → 시행판 20240701 · 조문 341개
현행                          → 시행판 20260101 · 조문 393개
```

## 무경고 함정을 가드로 막는다

```
law&LM=소득세법&efYd=20230101 → 시행일 20260101 · 조문 393개   ← efYd 가 조용히 무시됐다
```

`target=law` 는 `efYd` 를 **무시하고 현행을 준다. 경고도 없다.** 이걸 모르고 짜면 "2023년
지정했는데 2026년 조문"이 그대로 나간다. `assertEffectiveDateTarget` 이 `law`·`elaw`·
`lsDelegated` 로 시점 조회하는 경로를 즉시 실패시킨다 — **조용한 오답보다 시끄러운 실패가 낫다.**

## 추정하지 않는다

- 해석 못 하는 시점("작년", "최근", "2023년 1기")은 **null** — 호출자가 명시 거절해야 한다.
  현행으로 대체하면 닫는 기준 3("조용한 현행 반환 0건")이 무너진다.
- 요청 시점에 아직 없던 법도 **null** — 가장 오래된 판으로 대체하지 않는다.
- 연도만 주면 12/31 기준으로 보되, **그 가정을 규칙 문자열로 함께 낸다**. 법인세는 사업연도가
  역년과 다를 수 있어 12/31 가정이 틀릴 수 있고, 그 사실을 사용자가 검증할 수 있어야 한다.
