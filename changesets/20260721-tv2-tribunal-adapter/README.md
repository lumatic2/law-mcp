## 무엇이 달라지나

**세법 실무에서 실제로 다투는 곳이 처음으로 도구에 들어왔다.** 조세 분쟁은 법원에 가기 전에
조세심판원에서 결정된다 — "가산세" 하나로 4,688건, "1세대 1주택" 3,885건.

```
search_legal_source(source="ttDecc", query="이행강제금")
  → 조심 2015지0238 / 조세심판원 / 2016.01.29
     "불법으로 증축한 건물에 대하여 이행강제금을 납부하였으므로 취득세 등을…"
get_legal_source(source="ttDecc", source_id="68860")
  → 재결요지: "청구인이 이 건 건축물에 대한 이행강제금을 납부하였다고 하더라도 이는…"
     이유: "1. 처분개요 가. 청구인은 2002.8.20. …"
```

`재결요지` 는 판례의 판시사항에 해당하는 요약이 **upstream 에 이미 있는 것**이다 — 우리가
요약을 만들지 않는다(`nlrc` 판정요지와 같은 구조).

| source | 자료원 | 프로브 건수 |
|---|---|---|
| `ttDecc` | 조세심판원 결정례 | **4,688** ("가산세") |
| `adapDecc` | 소청심사 결정례 | 201 |
| `acrDecc` | 감사원 심사청구 결정례 | 79 |

도구 개수는 **11개 그대로**다(`source` enum 14 → 17). UD3 선례 — 자료원을 도구로 풀면 표면이
폭발하고 소비 LLM 의 선택 부담이 곧 품질 저하다.

## 명명 규칙이 세 군데에서 깨진다 — 전부 테스트로 고정했다

이 자료원은 "틀리면 조용히 실패"하는 함정이 유독 많다.

**① target 어순이 공식 문서와 반대다.** 문서는 `specialDeccTt` 로 적었는데 그 이름은 빈 응답이고,
실제로 응답하는 것은 `ttSpecialDecc`(기관약어 + 서비스명)다. 선행 조사
(`lawgo-api-survey.md` §D-2)가 이 때문에 "존재하지 않거나 파라미터 미상"으로 분류했다.
테스트가 `/^special/i` 를 거부한다 — **문서를 보고 되돌리는 것을 막는 장치**다.

**② 컨테이너·행 키가 target 에서 유도되지 않는다.** `committee()` 헬퍼는
`container = 첫 글자 대문자(target)` · `rowKey = target` 을 가정하는데, 이 3종은
target 과 무관하게 `Decc`/`decc` 다. 헬퍼를 쓰면 조용히 0건이 되므로 명시 descriptor 로 뒀다.

**③ 단건 ID 가 행의 `id` 가 아니다.** `id` 는 행 번호(1,2,3…)이고 실제 ID 는
`특별행정심판재결례일련번호` 다. 프로브에서 `ID=1` 로 부르자 무관한 문서가 200 으로 왔다.

추가로 **`type=HTML` 을 쓰지 않는다.** IB1b 에서 NTS 판례에 HTML 폴백을 붙인 선례가 있어
재사용하기 쉬운데, 여기서 HTML 은 본문 없는 JS 로더 셸(1,973B)이다. 테스트가 descriptor 에
`html` 문자열이 들어오는 것을 거부한다.

## `decc` 와 `ttDecc` 는 다른 자료원이다

이름이 비슷해 묶고 싶어지지만 **별도 컨테이너**다. 지금까지 조세가 사각지대였던 이유가 이것이다 —
`decc`(행정심판재결례)에는 조세 결정례가 없다. 테스트로 분리를 고정했다.
