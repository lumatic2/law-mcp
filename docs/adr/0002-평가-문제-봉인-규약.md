# ADR 0002 — 평가 문제 봉인 규약

- Status: Accepted (2026-07-23)
- Milestone: TF2 step-2 (horizon `trap-free`)

## Context

도구를 고칠 때 점수를 보면서 고치면, 결국 **그 문제들만 잘 푸는 도구**가 된다. 점수는 오르는데
실제 사용자의 다른 질문은 그대로다. 이걸 가리려면 문제 일부를 떼어 두고 **한 번도 보지 않은
채로** 마지막에 딱 한 번 풀어봐야 한다 — 그게 봉인이다. 한 번 열면 그 문제들의 답을 우리가
알게 되므로 다시는 같은 용도로 못 쓴다.

지금 상태를 실측하면 이렇다:

- 코퍼스 124 레코드 / 94 topic 중 **미개봉 문제는 0개**다.
- `dev` 79건은 튜닝 대상이라 계속 봐 왔다.
- `holdout` 45건은 이미 전부 개봉됐다 — `golden-v2` 홀드아웃은 `uninterrupted-delegation`,
  `golden-tax` 홀드아웃은 `tax-vertical`, `golden-tax-agentic` 홀드아웃 10건은 `agentic-reach`
  에서 열었다(`archive/reports/` 각 close 노트).

그리고 지난 horizon 은 홀드아웃이 소진될 때마다 **급히 새 문제를 만들어** 보충했다. 그 급조가
`agentic-reach` 오염의 직접 원인이다 — 10건 중 6건을 에이전트가 주제까지 골랐고, 고를 때 도구로
존재를 확인했다. 유출 탐지기는 문자열만 보므로 이 "구성에 의한 오염"을 못 잡았다.

## Decision

**봉인은 데이터를 새로 만들어 확보하는 것이 아니라, 문제를 늘릴 때 그 자리에서 떼어 두는 것이다.**

1. **다음에 문제를 늘릴 때, 늘리는 분량의 일부를 처음부터 `sealed` 로 표시하고 아무도 보지
   않는다.** 늘리는 계기(새 분야 vertical 등)에 얹어서 확보한다 — 봉인을 위해 따로 문제를
   만들지 않는다. 따로 만들기 시작하면 급조가 되고, 급조는 오염된다.
2. **주제는 사람이 정하거나 이미 있는 라벨에서 온다.** 에이전트가 주제를 고르지 않는다.
   이 규약은 `bench/check-no-new-topics.ts` 가 기계로 강제한다 — 코퍼스의 모든 정답 조문이
   통합 전 세트에 이미 있었음을 검증하고, 규약 적용으로 바뀐 라벨만 `label_rule` 표시로 예외.
3. **개봉은 기록으로 남기고 되돌리지 않는다.** 개봉한 문제는 그 뒤로 dev 로 강등해 계속 쓰되,
   과적합 판정 근거로는 다시 쓰지 않는다.
4. **봉인을 강제하는 코드는 미개봉 문제가 실제로 생길 때 만든다.** 지금 만들면 담을 것이 없어
   장치만 놀고, 놀고 있는 장치는 다음 사람이 신뢰한다(있으니까 지켜지고 있겠거니).
   현행 `assertHoldoutSeal` 은 `--split holdout` 을 플래그 없이 거절하는 상태로 유지한다.

## Consequences

- **이번 horizon 은 새 홀드아웃 수치를 내지 않는다.** 낼 수 없다 — 미개봉 문제가 0개다.
  대신 다음에 문제를 늘릴 때 이 규약이 작동해 자를 다시 세울 수 있다.
- 그래서 `trap-free` 의 닫는 기준에는 홀드아웃 수치가 없다. `dev` 재기준선과 주제 무개입
  증명만 담는다 — 없는 판정을 있는 척하지 않는다.
- 다음 horizon 이 분야를 늘리면(노동·부동산 등) 그 시점에 ① 늘리는 분량의 일부를 `sealed` 로
  표시 ② 봉인 강제 코드 작성 ③ 개봉 이력 파일 도입을 함께 한다.

## Alternatives considered

- **이번에 새 문제를 만들어 봉인을 채운다** — 급조가 오염을 부른 직전 실패를 그대로 반복한다.
  TF2 의 non-goals 에 "새 케이스 추가 금지"를 넣은 이유다.
- **`dev` 에서 일부를 떼어 봉인한다** — 이미 튜닝에 쓴 문제라 봉인해도 과적합 판정력이 없다.
  분모만 줄이고 얻는 게 없다.
